## 1. なぜこの章を書くのか

第7章では、Lido CSM独自の監視サービス「CSM Sentinel」が、スラッシングやExitのような「イベント系」には反応するものの、Inactivity Leakのような「状態系の劣化」には無反応であることを発見しました。第8章では、この課題への対策として、`node_check.sh`（プル型・手動確認）とGrafanaダッシュボード（プル型・可視化）を整備しました。

しかし、これらはいずれも**「自分から見に行く」プル型**の仕組みです。誰かが気づいて確認しない限り、異常は放置され続けます。本章では、**「異常があれば、向こうから知らせてくる」プッシュ型**の仕組みを、Grafana Alertingで実装します。

---

## 2. なぜGrafana Alertingを選んだか

第7章では、死活監視強化の候補として以下を挙げていました。

```
候補①：Grafana Alerting
候補②：Uptime Kuma
候補③：自作Cron + beaconcha.in API + Telegram Bot
```

第8章でPrometheus + Grafanaの基盤を構築済みであり、PromQLクエリの書き方・ラベル名の確認方法・`increase()`の使い方等のノウハウも蓄積されていたため、**新しいツールを別途導入する必要がない、最も自然な延長**としてGrafana Alertingを選びました。

```
✅ 無料：Grafana OSS版に標準で含まれる
✅ 拡張性：既存のPrometheus jobをそのまま流用できる
✅ 可視化（ダッシュボード）と通知（Alerting）が、同じPromQL
   クエリを土台に地続きで作れる
```

## 3. Telegram通知の準備

CSM Sentinelの通知と区別するため、Grafana専用の新しいTelegram Botを作成しました。

```
手順：
① Telegramで「@BotFather」を検索し、/newbot コマンドでBotを作成
② Bot名・ユーザー名を設定し、Bot API Token を取得
③ 作成したBotとのチャットでメッセージを送信
④ https://api.telegram.org/bot<TOKEN>/getUpdates にアクセスして
   Chat ID を取得
```

> 💡 **既存のCSM Sentinelとは別のBotを用意した理由：** 同じBotを共有すると、
> 「どちらからの通知か」が分かりにくくなります。監視レイヤーごとに
> 通知元を分けておくことで、後から見返したときの判別がしやすくなります。

Grafanaの「Notification configuration」→「Contact points」で、Telegram統合のコンタクトポイント（`lido-csm-telegram`）を作成し、Bot API Token・Chat IDを入力。テスト通知が正しく届くことを確認しました。

## 4. アラートルールの設計：3階層で監視する

第7章の課題（地味な劣化に気づけない）に応えるため、**「結果」「中間的な異常」「根本原因」の3つの異なる階層**から、同じ問題を多角的に検知する設計にしました。

```
階層①：根本原因（プロセス自体が落ちている）
  → Service Down (Geth/Lighthouse)

階層②：中間的な異常（プロセスは動いているが、同期していない）
  → Beacon Node Not Synced

階層③：結果としての劣化（実際に署名が失敗し始めている）
  → Attestation Success Rate Low
```

> 💡 **なぜ1つだけでなく3つ作ったのか：** 単一の指標だけでは、検知の
> タイミングや感度に偏りが出ます。根本原因（プロセス停止）を最速で
> 検知しつつ、もしそれを見逃しても中間指標・結果指標がバックアップ
> として機能する、多層防御の発想です。

### 4-1. Attestation Success Rate Low（結果を検知）

```yaml
名前: Attestation Success Rate Low
クエリ: rate(vc_signed_attestations_total{status="success"}[5m]) /
        rate(vc_signed_attestations_total[5m]) * 100
条件: IS BELOW 95
評価グループ: lido-csm-group（1分ごと）
保留期間: 1m
データなしの場合: Alerting
実行エラーの場合: Alerting
Summary: ⚠️ Attestation成功率が低下しています
Description: 直近5分間のAttestation成功率が低下しています。
             node_checkまたはGrafanaダッシュボードで詳細を確認してください。
```

### 4-2. Beacon Node Not Synced（中間的な異常を検知）

```yaml
名前: Beacon Node Not Synced
クエリ: vc_beacon_nodes_synced_count
条件: IS BELOW 1
データなしの場合: Alerting
実行エラーの場合: Alerting
Summary: 🚨 Beacon Nodeが同期していません
Description: lighthouse_vcが認識しているBeacon Nodeの同期数が0になりました。
             geth/lighthouseのサービス状態を直ちに確認してください。
```

### 4-3. Service Down (Geth/Lighthouse)（根本原因を検知）

```yaml
名前: Service Down (Geth/Lighthouse)
クエリ: min(up{job=~"geth|lighthouse_bn|lighthouse_vc"})
条件: IS BELOW 1
データなしの場合: Alerting
実行エラーの場合: Alerting
Summary: 🔴 geth/lighthouseサービスが停止しています
Description: geth・lighthouse_bn・lighthouse_vcのいずれかで応答が
             取れていません。node_checkでサービス状態を直ちに
             確認してください。
```

> 💡 **`up`メトリクスについて：** Prometheusは、スクレイプ対象に
> 接続できたかどうかを示す標準メトリクス`up`を、すべてのjobに対して
> 自動的に生成します（1=応答あり、0=応答なし）。これを使えば、
> 個別のアプリケーションメトリクスに依存せず、プロセスの生死を
> 直接判定できます。`min(up{job=~"geth|lighthouse_bn|lighthouse_vc"})`
> という書き方で、3つのうち1つでも落ちていれば検知できます
> （geth・lighthouse_bn・lighthouse_vcは、どれか1つが落ちれば全体が
> 機能しなくなる依存関係にあるため、まとめて1つのルールで監視する
> 設計にしました）。

### 4-4. 「データなし」設定の重要性

すべてのルールで、「データがないか、すべての値がnullの場合のアラート
ステート」を、デフォルトの`No Data`から`Alerting`に変更しました。

```
理由：
  サーバーが完全に停止すると、Prometheusはそもそもメトリクスを
  取得できなくなる（No Data状態になる）
  → この設定を変更しないと、「成功率が95%未満」という条件評価
    自体が行われず、最も深刻な「完全停止」のケースを
    アラートが見逃してしまう
```

これは、第8章で発見した「累積カウンターは単発確認に向かない」という
教訓とは別の、もう一つの重要な落とし穴でした。**メトリクスの「値」だけ
でなく「値が取得できない」状態そのものも、異常として扱う必要がある**
ということです。

## 5. 実機検証：実際にサービスを停止して確認する

設計したアラートルールが実際に機能するか、`node_safe_stop.sh`を使って
実際にサービスを停止し、検証しました。

### 5-1. 検証前のベースライン確認

```
node_checkで全項目が正常であることを確認：
  geth/lighthouse/mev-boost/lighthouse-vc：すべてactive
  is_syncing: false, sync_distance: 0
  10バリデータすべてactive_ongoing
  SSV Node：すべて健全
```

### 5-2. node_safe_stop.shの実行

```
1. Stopping SSV Node (DVT operator)...
 ✔ Container ssv-node Stopped
2. Stopping Validator Client (lighthouse-vc)...
3. Stopping Beacon Node and MEV-Boost...
4. Stopping Execution Client (Geth)...
[Verification] Check Service Status:
 - lighthouse-vc: inactive (Safe)
 - lighthouse: inactive (Safe)
 - mev-boost: inactive (Safe)
 - geth: inactive (Safe)
 - ssv-node: exited (Safe)
```

### 5-3. アラート発火のタイムライン（実測）

```
08:32 Firing: Beacon Node Not Synced
08:32 Firing: Service Down (Geth/Lighthouse)
08:37 Firing: Attestation Success Rate Low（5分遅れて発火）
```

3つのアラートすべてが正しく発火しました。Telegram通知の内容も、
設定した日本語の文言（Summary・Description）が正確に反映されていました。

```
**Firing**
alertname = Beacon Node Not Synced
grafana_state_reason = NoData
summary = 🚨 Beacon Nodeが同期していません
```

> 💡 **「Service Down」「Beacon Node Not Synced」が同時に、
> 「Attestation Success Rate Low」だけが5分遅れて発火した理由：**
> 前者2つは即時的な状態フラグ（`up`・`vc_beacon_nodes_synced_count`）
> を見ているため、サービス停止に即座に反応します。一方、後者は
> `rate(...)[5m]`という5分間の移動平均を使っているため、判定に
> 必然的にラグが生じます。これは第8章で扱った「即時指標」と
> 「期間集計指標」の特性差が、実際の検証で裏付けられた結果です。

### 5-4. サービス復帰と「Resolved」の確認

サービス停止と合わせて、OSアップデート（`sudo apt update && sudo apt
upgrade -y`）とサーバー再起動も実施しました（Grafana自体も13.0.2→
13.1.0にアップデートされましたが、アラートルールの設定は正しく
保持されていました）。

再起動後、`node_check`で全サービスの復帰を確認（`restart: always`の
設定により、SSVノードも手動操作なしで自動復帰）。その後、Telegramに
以下の「Resolved」通知が届きました。

```
08:41 Resolved: Beacon Node Not Synced
08:41 Resolved: Service Down (Geth/Lighthouse)
08:42 Resolved: Attestation Success Rate Low（1分遅れて解決）
```

発火時と全く同じ順序・同じ時間差のパターンで解決しており、
即時指標と期間集計指標の特性差が、発火・解決の両方で一貫して
再現されることが確認できました。

## 6. まとめ

| 知見 | 内容 |
|---|---|
| プル型とプッシュ型の組み合わせ | 第8章（プル型：node_check.sh・Grafanaダッシュボード）に、本章（プッシュ型：Grafana Alerting）を組み合わせることで、「自分で見に行く」「向こうから知らせてくる」の両方が揃った |
| 3階層の監視設計 | 根本原因（プロセス停止）・中間的な異常（同期切れ）・結果（成功率低下）を、それぞれ独立したルールで検知する多層防御 |
| `up`メトリクスの有用性 | アプリケーション固有のメトリクスに依存せず、Prometheus標準の`up`でプロセスの生死を直接判定できる |
| 「データなし」設定の重要性 | デフォルトの`No Data`のままでは、最も深刻な「完全停止」のケースを見逃す。`Alerting`への変更が必須 |
| 即時指標と期間集計指標の時間差 | `up`や状態フラグは即時に反応するが、`rate()`等の移動平均は5分程度のラグが生じる。これは発火時・解決時の両方で一貫して再現された |
| 実機検証の価値 | 設計しただけでは分からない「実際の発火順序」「タイムラグ」が、実際にサービスを止めてみることで初めて明らかになった |
