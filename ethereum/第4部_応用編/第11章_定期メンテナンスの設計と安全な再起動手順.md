## 1. 定期再起動が必要な理由

バリデータノードは原則として止めないことが理想です。しかし現実の運用では、
止めないことで生じる別のリスクが蓄積していきます。本章で扱うのはその代表的な
2つです。

### 1-1. Swapの蓄積

`node_check` の `[2] Resource Usage` を日次で観測していると、Swap使用量が
ゆっくりと増加していくことがあります。

```
6/28: Swap used 1.1Gi
7/01: Swap used 2.2Gi  ← 4日間で約1Gi増加
```

Linuxカーネルは、一度スワップアウトしたページを、メモリに余裕が戻っても
自動的に解放しません。これはバグではなく仕様です。つまり過去のある時点で
メモリ使用量がピークに達した際にスワップへ退避されたページが、そのまま
残り続けている状態です。

### 1-2. OSセキュリティアップデートの滞留

`node_check` の `[11] OS Update & Restart Status` に以下のような表示が
出ている場合、セキュリティパッチが未適用のまま稼働しています。

```
Pending Updates  : Updates Available!
    3 updates can be applied immediately.
```

適用自体はノードを止めずにできますが、カーネルアップデートが含まれる場合は
再起動が必要です。`Restart Required: YES` が出たまま放置すると、パッチが
適用されていない期間が長くなります。

## 2. スワップについての整理

### 2-1. 「スワップ無効化」を検討したが現状維持と判断した理由

Swap使用量が増加傾向にあることに気づき、スワップ無効化（`swapoff -a`）を
検討しました。しかし以下の理由から、**無効化はしない**という判断に至りました。

| 観点 | 内容 |
|---|---|
| メモリスパイクの安全弁 | Geth・Lighthouse・SSVノードは同期処理やDBコンパクション時に一時的なメモリ使用量の急増を起こすことがある。スワップがなければ、その瞬間にOOM Killerがプロセスをkillしにくる |
| 32GBという制約 | 複数の重量級プロセス（Geth・Lighthouse BN/VC・MEV-Boost・SSV）が同時稼働している環境では、スパイク時の緩衝材を完全に取り除くリスクが大きい |
| 高速NVMe | 高速NVMeを採用しているため、スワップが多少発生してもレイテンシへの影響は小さい |

### 2-2. vm.swappiness の確認と設定

スワップを保持しつつ「不要な先回り退避」を防ぐために、`vm.swappiness` を
確認します。

```bash
# なぜ：デフォルト（60）のままだと、メモリに余裕があるのに積極的にスワップを
# 使い始めてしまいます。現在の設定値を確認します
cat /proc/sys/vm/swappiness
```

デフォルト値の60は「メモリ使用率が40%を超えたあたりからスワップを使い始める」
積極的な設定です。バリデータ運用では**10**が推奨値です。

```
10 → 本当にメモリが逼迫する直前までスワップを使わない
```

すでに10が設定済みであれば、追加の対応は不要です。もし60のままであれば
以下で変更します。

```bash
# 一時的に変更（再起動で元に戻る。効果を確認するための試験用）
sudo sysctl vm.swappiness=10

# 永続化（再起動後も設定を保持する）
echo 'vm.swappiness=10' | sudo tee -a /etc/sysctl.conf
```

> 💡 **Swapが増えていても、available（利用可能メモリ）が十分にある限り
> パフォーマンス上の問題はありません。** 見た目が気になる場合は、再起動の
> 代わりに以下のコマンドでサービスを止めずにSwapだけクリアできます。
>
> ```bash
> # なぜ：Swap領域の定義はそのまま、中身だけメモリに書き戻します。
> # geth・lighthouse等のサービスは無停止で実行できます
> sudo swapoff -a && sudo swapon -a
> ```

## 3. 再起動前の安全確認：ブロック提案チェック

### 3-1. なぜブロック提案を確認するのか

Ethereumのブロック提案者（Proposer）は、エポック（約6.4分）単位で割り当て
られます。再起動のタイミングが自分のバリデータの提案スロットと重なると、
ブロック提案を逃してしまいます。

attestationミスと比較したブロック提案の機会損失：

```
attestationミス : 次のスロット（12秒後）に他のバリデータが挽回できる
ブロック提案ミス: そのスロット限りのチャンス。逃したら次の割り当てまで取り戻せない
```

### 3-2. 確認コマンド（手動実行）

```bash
# Step 1：現在のエポックを確認する
# なぜ：Proposer dutiesはエポック番号を引数に渡す必要があります
curl -s http://localhost:5052/eth/v1/beacon/headers/head \
  | python3 -c "
import sys, json
d = json.load(sys.stdin)
slot = int(d['data']['header']['message']['slot'])
epoch = slot // 32
print(f'現在のスロット: {slot}')
print(f'現在のエポック: {epoch}')
"
```

```bash
# Step 2：そのエポックの提案予定者一覧を取得する
# なぜ：1エポック（32スロット）分の提案予定者が一覧で返ってきます
# <EPOCH> には Step 1 で取得したエポック番号を入れてください
curl -s http://localhost:5052/eth/v1/validator/duties/proposer/<EPOCH> \
  | python3 -m json.tool
```

返ってくるJSONの読み方：

```json
{
    "data": [
        {
            "pubkey": "0xab23...",
            "validator_index": "1277372",
            "slot": "3390080"
        },
        ...
    ]
}
```

`validator_index` と `slot` の組み合わせが32件（1エポック分）表示されます。
この中に自分のバリデータインデックスが含まれていなければ、安全に停止できます。

### 3-3. 自分のバリデータインデックスの確認方法

pubkey（`node_check` の `[7]` で確認できる短縮形）から、インデックスを
取得します。

```bash
# なぜ：Proposer dutiesはインデックスで管理されているため、
# pubkeyではなくインデックスで突き合わせる必要があります
curl -s http://localhost:5052/eth/v1/beacon/states/head/validators \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)

# ★ここを自分のpubkey先頭8文字に書き換えてください（node_checkの[7]で確認）
my_pubkeys = [
    '0xXXXXXXXX',
    '0xYYYYYYYY',
]

for v in data['data']:
    pk = v['validator']['pubkey']
    for short in my_pubkeys:
        if pk.startswith(short):
            print(f\"index: {v['index']}  pubkey: {pk[:20]}...\")
" 2>&1 | head -20
```

> ⚠️ **このAPIは全バリデータ（数十万件）を返すため、レスポンスに数十秒
> かかる場合があります。** 一度確認したインデックスは手元に記録しておき、
> 次回以降はそのまま使えます。

## 4. node_safe_stop.sh へのブロック提案チェック組み込み

手動コマンドでの確認を毎回行うのは手間がかかるため、`node_safe_stop.sh` の
停止処理の直前に自動チェックを組み込みます。

### 4-1. スクリプトの冒頭に設定を追加

```bash
# ============================================================
# ★ここを自分の環境に書き換えてください★
# 自分のバリデータインデックスをスペース区切りで記入します。
# 確認方法はセクション3-3を参照してください。
# ============================================================
MY_VALIDATOR_INDICES="<自分のindex_1> <自分のindex_2> <自分のindex_3>"

BEACON_URL="http://localhost:5052"
```

### 4-2. Step 0 として停止処理の直前に挿入

```bash
# ============================================================
# Step 0: ブロック提案チェック（停止前の安全確認）
# 【Why】再起動のタイミングがブロック提案スロットと重なると、
# 提案を逃してしまいます。attestationミスより報酬への影響が
# 大きいため、停止の直前に必ず確認します。
# チェックできるのは現在のエポック（約6.4分先）までです。
# ============================================================
echo -e "\n${YELLOW}[Step 0] Checking Block Proposal Schedule...${NC}"

CURRENT_SLOT=$(curl -s "${BEACON_URL}/eth/v1/beacon/headers/head" \
  | python3 -c "
import sys, json
d = json.load(sys.stdin)
print(int(d['data']['header']['message']['slot']))
" 2>/dev/null)

if [ -z "$CURRENT_SLOT" ]; then
    echo -e " ${RED}⚠ Beacon NodeへのAPIアクセスに失敗しました。${NC}"
    echo -e " ${YELLOW}beaconcha.inで手動確認してから続行してください。続行しますか？ (y/N): ${NC}"
    read -r ANSWER
    if [[ "$ANSWER" != "y" && "$ANSWER" != "Y" ]]; then
        echo -e "${GREEN}停止をキャンセルしました。${NC}"
        exit 0
    fi
else
    CURRENT_EPOCH=$((CURRENT_SLOT / 32))
    echo -e " 現在のスロット: ${CURRENT_SLOT}（エポック: ${CURRENT_EPOCH}）"

    DUTIES=$(curl -s "${BEACON_URL}/eth/v1/validator/duties/proposer/${CURRENT_EPOCH}" 2>/dev/null)

    PROPOSAL_FOUND=$(echo "$DUTIES" | python3 -c "
import sys, json
data = json.load(sys.stdin)
my_indices = set('${MY_VALIDATOR_INDICES}'.split())
found = []
for d in data.get('data', []):
    if d['validator_index'] in my_indices:
        found.append(f\"  ⚠ スロット {d['slot']} にインデックス {d['validator_index']} の提案予定あり\")
for f in found:
    print(f)
" 2>/dev/null)

    if [ -n "$PROPOSAL_FOUND" ]; then
        echo -e "\n ${RED}⚠ WARNING: 現在のエポック内にブロック提案が予定されています！${NC}"
        echo -e "$PROPOSAL_FOUND"
        echo -e "\n ${YELLOW}このまま停止するとブロック提案を逃します。${NC}"
        echo -e " ${YELLOW}約6.4分後（次のエポック開始後）に再実行することを推奨します。${NC}"
        echo -e " ${YELLOW}それでも今すぐ停止しますか？ (y/N): ${NC}"
        read -r ANSWER
        if [[ "$ANSWER" != "y" && "$ANSWER" != "Y" ]]; then
            echo -e "${GREEN}停止をキャンセルしました。次のエポック開始後に再実行してください。${NC}"
            exit 0
        fi
        echo -e " ${YELLOW}ユーザー確認済み。停止処理を続行します。${NC}"
    else
        echo -e " ${GREEN}✓ 現在のエポックにブロック提案の予定はありません。安全に停止できます。${NC}"
    fi
fi
```

> 💡 **「提案あり」と表示されたときの対応：**
> スクリプトがキャンセルされたら、約6.4分（次のエポック開始）待って
> 再実行します。最悪でも「6.4分待って再試行」で済むため、実用上の
> 支障はほとんどありません。

## 5. 定期再起動の実施手順

週1回（例：月曜または火曜の朝JST）を定例として実施します。

### Step 1：node_check で現状確認

```bash
node_check
```

確認ポイント：

| 項目 | 確認内容 |
|---|---|
| `[1] System Services` | 全サービスが `active` であること |
| `[2] Swap used` | 蓄積量の把握（再起動後との比較用に記録） |
| `[11] Pending Updates` | アップデートの有無と件数を確認 |

### Step 2：安全停止スクリプトの実行

```bash
# なぜ：ブロック提案チェック → SSVノード → lighthouse-vc →
# lighthouse → mev-boost → geth の順で安全に停止します
./node_safe_stop.sh
```

ブロック提案が検出された場合はスクリプトが自動的に止まります。
`N` を選択してキャンセルし、約6.4分後に再実行してください。

### Step 3：OSアップデートの適用（Pending Updatesがある場合のみ）

```bash
# なぜ：セキュリティパッチを適用します。
# node_check の [11] に「Updates Available!」が出ている場合のみ実施します
sudo apt update && sudo apt upgrade -y
```

### Step 4：再起動

```bash
# なぜ：Swap領域のクリア・カーネルアップデートの反映を行います
sudo reboot
```

### Step 5：再起動後の正常確認

再起動後、SSHで再接続して `node_check` を実行します。

```bash
node_check
```

確認ポイント：

| 項目 | 期待値 |
|---|---|
| `[1] System Services` | 全サービスが `active` |
| `[2] Swap used` | 再起動直後は `0` または大幅に減少 |
| `[3] Sync Status` | `is_syncing: false` |
| `[5] Recent Attestations` | `Successfully published attestations` が出ていること |
| `[11] Restart Required` | `NO` |
| `[12] SSV Node` | `attesting` |

> ⚠️ **再起動直後はGethとLighthouseの起動順序に注意してください。**
> Gethは単独では同期できず、Lighthouseが起動してから同期を開始します
> （Post-Merge アーキテクチャの仕様）。サービスはsystemdの自動起動に
> 任せ、`node_check` で `is_syncing: false` が返るまで数分待ちます。

## 6. まとめ

| 知見 | 内容 |
|---|---|
| 定期再起動の目的は2つ | Swapの蓄積解消 と OSセキュリティアップデートの適用 |
| スワップは無効化しない | バリデータ運用ではスパイク時の安全弁として有効。vm.swappiness=10 で「先回り退避」を抑えつつ保持する |
| Swap使用量の増加は正常 | メモリ余裕があっても過去のスパイク時のページが残り続ける。available が十分なら実害なし |
| ブロック提案は逃せない | attestationミスと違い、提案を逃すと次の割り当てまで取り戻せない。停止前の確認は必須 |
| チェックできるのは現在のエポックのみ | 提案スケジュールは約6.4分先（1エポック）までしか分からない。スクリプトは「今まさに提案が当たっているか」を確認する安全弁 |
| 提案ありでも最大6.4分待てばよい | キャンセル後、次のエポック開始を待って再実行するだけ。実用上の支障はほとんどない |
| apt upgrade は停止後に実施 | サービス停止後・再起動前のタイミングが最も安全 |
| 再起動後は node_check で必ず確認 | 自動起動が正しく機能しているか、attestationが再開されているかを目視確認する |
