## 1. スラッシングとは何か

スラッシング（Slashing）とは、バリデータが不正な署名を行った場合に課されるEthereumのペナルティ制度です。

### 二重署名（Double Voting）とは

バリデータは各スロットで「このブロックが正しい」という署名（アテステーション）を1回だけ行う義務があります。同じスロットに対して矛盾する署名を2回提出することを**二重署名**と呼び、これがスラッシングの主な原因です。

```
正常な状態：
  バリデータA → slot 100 に署名（1回）✅

二重署名：
  バリデータA → slot 100 に署名（1回目）
  バリデータA → slot 100 に別の内容で署名（2回目）← これがスラッシング
```

### スラッシングのペナルティ

| ペナルティ | 内容 |
|---|---|
| Initial Penalty | デポジットの1/32を即座に没収 |
| Correlation Penalty | 同時期に不正を行ったバリデータが多いほど増加 |
| Forced Exit | 一定期間後に強制退出（Withdrawal Queueに追加） |

Lido CSMでは、スラッシングが発生するとボンド（担保）が削減されます。

### 「即バン」ではない：スラッシングの仕組み

スラッシングは二重署名した瞬間に即座に発動するわけではありません。

```
① バリデータが二重署名を送信
        ↓
② ネットワーク上の「スラッシャー」が証拠を収集
   （他のバリデータノードが両方の署名データを集める）
        ↓
③ 証拠が「告発ブロック」に組み込まれて確定
        ↓
④ ペナルティが執行・beaconcha.inに反映
```

**スラッシャー**とは、二重署名の証拠を収集して告発ブロックに含めるノードです。
告発に成功すると報酬が得られるため、バウンティハンターのような役割を担います。

> 💡 **テストネットでは検知が遅い場合があります。**
> 本番メインネットではスラッシャーが多く比較的早く検知されますが、
> Hoodi testnetは参加者が少ないため、検知まで数時間かかることがあります。

## 4. 検証手順：意図的な二重起動

### この検証の構成：なぜtestnetcsm2を止めないのか

この検証では第1章で構築した2台のVMをそのまま使います。

| VM | 状態 | 役割 |
|---|---|---|
| testnetcsm2 | **稼働中・署名中** | 止めない（復元元に相当） |
| testnetcsm | **停止中** | これを起動する（復元先に相当） |

**testnetcsm2を止めずにtestnetcsmを起動します。**

これは第1章で警告した「復元元と復元先の同時稼働」を意図的に再現した状態です。
【なぜこの構成なのか】
testnetcsm2を止めてからtestnetcsmを起動する

→ これは「引越し」= スラッシングは発生しない
testnetcsm2を止めずにtestnetcsmを起動する

→ 2台が同じ鍵で同時稼働 = 二重署名 = スラッシング発生

↑ 今回はこれを体験する

> ⚠️ **本番環境では絶対に行わないでください。**
> この検証はHoodi Testnetのテスト用ETHを使った安全な環境でのみ実施しています。

### testnetcsmのサービスを起動する

```bash
# ① サービスをenableに戻す
# disable済みだったサービスを自動起動対象に戻す
sudo systemctl enable geth lighthouse lighthouse-vc

# ② 順番通りに起動する
# Geth（実行レイヤー）を先に起動する
sudo systemctl start geth

# Lighthouse BN（合意レイヤー）を起動する
sudo systemctl start lighthouse

# Lighthouse VC（バリデータクライアント）を起動する
sudo systemctl start lighthouse-vc
```

### なぜすぐには二重署名にならないのか

起動直後、lighthouse-vcのログには以下が出ます：

```
INFO  All validators inactive   validators: 2, epoch: 101634
ERROR No synced beacon nodes    total: 1, available: 0, synced: 0
```

**VCはBN（Beacon Node）が同期完了するまで署名できません。**

```
起動直後の状態：
  Lighthouse BN: 同期中（チェーンに追いついていない）
  Lighthouse VC: BNの同期完了を待機中
  → この間はtestnetcsm2との二重署名は発生しない
```

BNの同期完了まで待つ必要があります。

### Lighthouse BNの同期を確認する

```bash
# BNの同期進捗を確認する
sudo journalctl -u lighthouse --no-pager | grep "Syncing\|Synced" | tail -3
```

**ログの読み方：**

```
INFO  Syncing  peers: "25", distance: "4807 slots (16 hrs 1 mins)",
               speed: "2.00 slots/sec", est_time: "40 mins"
```

| 項目 | 意味 |
|---|---|
| `peers` | 接続中のピア数。多いほど同期が安定 |
| `distance` | チェーン先端までの遅れ（スロット数） |
| `speed` | 同期速度（slots/sec） |
| `est_time` | 同期完了までの推定時間 |

今回の実測値：

| 時刻（UTC） | distance | est_time | peers |
|---|---|---|---|
| 05:14 | 7,206 slots | 1hr 46mins | 8 |
| 05:19 | 6,652 slots | 54mins | 16 |
| 05:30 | 5,330 slots | 43mins | 23 |
| 06:04 | 1,243 slots | 7mins | 25 |
| 06:15 | 0 slots | - | 25 |

**約1時間かけてBNの同期が完了しました。**

### Gethの同期確認

```bash
# Gethの同期状態を確認する
curl -s http://127.0.0.1:8545 \
  -X POST \
  -H "Content-Type: application/json" \
  --data '{"jsonrpc":"2.0","method":"eth_syncing","params":[],"id":1}' | jq
```

**結果の読み方：**

| 結果 | 意味 |
|---|---|
| `"result": false` | 同期完了 ✅ |
| `"result": { ... }` | 同期中（数値が表示される） |

今回の実測：BN同期完了の約45分後に `"result": false` を確認。

## 5. 二重署名発生の記録（実ログ）

### BN同期完了・VCがアクティブになった瞬間

```
Jun 12 06:15:42 testnetcsm lighthouse[1190]:
  INFO  Synced
  peers: "25", finalized_epoch: 101642, epoch: 101644, slot: 3252628

Jun 12 06:16:06 testnetcsm lighthouse[1220]:
  INFO  All validators active
  current_epoch_proposers: 0, active_validators: 2,
  total_validators: 2, epoch: 101644, slot: 3252630
```

**この瞬間から、testnetcsmとtestnetcsm2が同じ鍵で同時に署名を開始しました。**

### 決定的な二重署名の瞬間

**2台が同じslotで同時に署名した記録：**

```
【testnetcsm2】
Jun 12 06:28:25 testnetcsm2 lighthouse[1101]:
  INFO  Successfully published attestations
  count: 1, validator_indices: [1412113],
  slot: 3252692

【testnetcsm】
Jun 12 06:28:25 testnetcsm lighthouse[1220]:
  INFO  Successfully published attestations
  count: 1, validator_indices: [1412113],
  slot: 3252692
```

**同じ時刻・同じvalidator（1412113）・同じslot（3252692）で両方が署名を送信しました。これが二重署名です。**

### 重要な発見①：Lighthouseのログには警告が出ない

二重署名が発生しても、**Lighthouseのログには `Slashable` などの警告は出ませんでした。**

```
理由：
Lighthouseのスラッシング保護は
「自分自身の過去の署名DBと照合する」仕組み

testnetcsmのDB = 第1章の復元時点のスナップショット
testnetcsm2のDB = リアルタイム更新中

→ 2台はお互いの署名を知らない
→ 自分のDBと照合しても矛盾しない
→ 警告が出ないまま署名し続ける
```

**スラッシング保護DBは「同一マシン内での二重署名」を防ぐものです。別マシンが同時に署名することは防げません。**

### 重要な発見②：「即バン」ではない

二重署名から1時間以上経過しても、beaconcha.inのStatusは `Active` のままでした。

```
スラッシングの検知は即座ではない
  ↓
ネットワーク上のスラッシャーが
証拠（両方の署名データ）を収集する必要がある
  ↓
証拠が告発ブロックに組み込まれて初めて執行される
  ↓
テストネットではスラッシャーが少ないため
検知まで数時間かかることがある
```

### 重要な発見③：BeaconScoreが一時的に改善した

二重稼働によりアテステーション数が増加したため、BeaconScoreが一時的に改善しました。

| 時刻 | BeaconScore | 備考 |
|---|---|---|
| 14:11 JST（事前） | 23.93% | testnetcsm2のみ稼働 |
| 15:33 JST（Last 1h） | 92.16% | 両VM稼働・アテステーション増加 |
| 15:35 JST（Last 24h） | 27.82% | 24h全体で見ると低下中 |

**「スコアが上がった＝安全」ではありません。二重署名の証拠が集まり次第スラッシングが執行されます。**

### 各バリデータの状態推移

| 時刻（JST） | validator 1412113 | validator 1408787 |
|---|---|---|
| 14:11（事前） | Active / pending前 | Active / pending前 |
| 15:36 | Active / epoch 101,644〜 pending | Active / epoch 101,645〜 pending |
| 15:39 | Active / Balance 32.00008 ETH | Active / Balance 32.00028 ETH |
| 16:02 | Active / epoch 101,648〜 pending | Active / epoch 101,648〜 pending |

### `pending...` の意味

二重署名開始以降のepochが `pending...` 表示になっています。

```
pending... = 「報酬の計算が確定していない」状態

考えられる理由：
  ① ネットワークがこの署名の正当性を検証中
  ② スラッシング処理の前段階として保留中
  ③ beaconcha.inの集計処理待ち
```

## 8. まとめ：今回の検証で得られた知見

| 知見 | 内容 |
|---|---|
| スラッシングは即座ではない | 二重署名後もしばらくActiveのまま。スラッシャーが証拠を集めてブロックに含めるまで時間がかかる |
| ログには警告が出ない | Lighthouseは別マシンの署名を知らない。ログに `Slashable` は出ない |
| 保護DBは万能ではない | スラッシング保護DBは「自分自身の過去の署名」との照合のみ。別マシンの同時署名は防げない |
| テストネットは検知が遅い | スラッシャーが少ないため、本番より検知に時間がかかる |
| BeaconScore一時改善 | 二重稼働でアテステーション数が増えスコアが上がる。スコア改善≠安全 |

### 本番環境で絶対に守るべきこと

```
【第1章との繋がり】

復元作業の最大リスク =「復元元と復元先の同時稼働」

今回の検証で体験したこと：
  2台が同時に稼働 → 二重署名 → スラッシング（執行待ち）

守るべき手順：
  ① 旧環境のサービスを完全停止・確認
  ② 停止確認後に新環境を起動
  ③ 絶対に同時稼働させない
```
