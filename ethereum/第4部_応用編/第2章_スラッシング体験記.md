## 1. スラッシングとは何か

スラッシング（Slashing）とは、バリデータが不正な署名を行った場合に課されるEthereumのペナルティ制度です。

### 二重署名（Double Voting）とは

バリデータは各スロットで「このブロックが正しい」という署名（アテステーション）を1回だけ行う義務があります。同じスロットに対して矛盾する署名を2回提出することを**二重署名**と呼び、これがスラッシングの主な原因です。

```
正常な状態：
  バリデータA → slot 100 に署名（1回）✅

二重署名：
  バリデータA → slot 100 に署名（1回目）
  バリデータA → slot 100 に別の内容で署名（2回目）← これがスラッシング
```

### スラッシングのペナルティ

| ペナルティ | 内容 |
|---|---|
| Initial Penalty | デポジットの1/32を即座に没収 |
| Correlation Penalty | 同時期に不正を行ったバリデータが多いほど増加 |
| Forced Exit | 一定期間後に強制退出（Withdrawal Queueに追加） |

Lido CSMでは、スラッシングが発生するとボンド（担保）が削減されます。

### 「即バン」ではない：スラッシングの仕組み

スラッシングは二重署名した瞬間に即座に発動するわけではありません。

```
① バリデータが二重署名を送信
        ↓
② ネットワーク上の「スラッシャー」が証拠を収集
   （他のバリデータノードが両方の署名データを集める）
        ↓
③ 証拠が「告発ブロック」に組み込まれて確定
        ↓
④ ペナルティが執行・beaconcha.inに反映
```

**スラッシャー**とは、二重署名の証拠を収集して告発ブロックに含めるノードです。
告発に成功すると報酬が得られるため、バウンティハンターのような役割を担います。

> 💡 **テストネットでは検知が遅い場合があります。**
> 本番メインネットではスラッシャーが多く比較的早く検知されますが、
> Hoodi testnetは参加者が少ないため、検知まで数時間かかることがあります。

## 4. 検証手順：意図的な二重起動

### この検証の構成：なぜtestnetcsm2を止めないのか

この検証では第1章で構築した2台のVMをそのまま使います。

| VM | 状態 | 役割 |
|---|---|---|
| testnetcsm2 | **稼働中・署名中** | 止めない（復元元に相当） |
| testnetcsm | **停止中** | これを起動する（復元先に相当） |

**testnetcsm2を止めずにtestnetcsmを起動します。**

これは第1章で警告した「復元元と復元先の同時稼働」を意図的に再現した状態です。

```
【なぜこの構成なのか】

testnetcsm2を止めてからtestnetcsmを起動する
→ これは「引越し」= スラッシングは発生しない

testnetcsm2を止めずにtestnetcsmを起動する
→ 2台が同じ鍵で同時稼働 = 二重署名 = スラッシング発生
                                   ↑ 今回はこれを体験する
```

> ⚠️ **本番環境では絶対に行わないでください。**
> この検証はHoodi Testnetのテスト用ETHを使った安全な環境でのみ実施しています。

### testnetcsmのサービスを起動する

```bash
# ① サービスをenableに戻す
# disable済みだったサービスを自動起動対象に戻す
sudo systemctl enable geth lighthouse lighthouse-vc

# ② 順番通りに起動する
# Geth（実行レイヤー）を先に起動する
sudo systemctl start geth

# Lighthouse BN（合意レイヤー）を起動する
sudo systemctl start lighthouse

# Lighthouse VC（バリデータクライアント）を起動する
sudo systemctl start lighthouse-vc
```

### なぜすぐには二重署名にならないのか

起動直後、lighthouse-vcのログには以下が出ます：

```
INFO  All validators inactive   validators: 2, epoch: 101634
ERROR No synced beacon nodes    total: 1, available: 0, synced: 0
```

**VCはBN（Beacon Node）が同期完了するまで署名できません。**

```
起動直後の状態：
  Lighthouse BN: 同期中（チェーンに追いついていない）
  Lighthouse VC: BNの同期完了を待機中
  → この間はtestnetcsm2との二重署名は発生しない
```

BNの同期完了まで待つ必要があります。

### Lighthouse BNの同期を確認する

```bash
# BNの同期進捗を確認する
sudo journalctl -u lighthouse --no-pager | grep "Syncing\|Synced" | tail -3
```

**ログの読み方：**

```
INFO  Syncing  peers: "25", distance: "4807 slots (16 hrs 1 mins)",
               speed: "2.00 slots/sec", est_time: "40 mins"
```

| 項目 | 意味 |
|---|---|
| `peers` | 接続中のピア数。多いほど同期が安定 |
| `distance` | チェーン先端までの遅れ（スロット数） |
| `speed` | 同期速度（slots/sec） |
| `est_time` | 同期完了までの推定時間 |

今回の実測値：

| 時刻（UTC） | distance | est_time | peers |
|---|---|---|---|
| 05:14 | 7,206 slots | 1hr 46mins | 8 |
| 05:19 | 6,652 slots | 54mins | 16 |
| 05:30 | 5,330 slots | 43mins | 23 |
| 06:04 | 1,243 slots | 7mins | 25 |
| 06:15 | 0 slots | - | 25 |

**約1時間かけてBNの同期が完了しました。**

### Gethの同期確認

```bash
# Gethの同期状態を確認する
curl -s http://127.0.0.1:8545 \
  -X POST \
  -H "Content-Type: application/json" \

**結果の読み方：**

## 5. 二重署名発生の記録（実ログ）

### BN同期完了・VCがアクティブになった瞬間

```
Jun 12 06:15:42 testnetcsm lighthouse[1190]:
  INFO  Synced
  peers: "25", finalized_epoch: 101642, epoch: 101644, slot: 3252628

Jun 12 06:16:06 testnetcsm lighthouse[1220]:
  INFO  All validators active
  current_epoch_proposers: 0, active_validators: 2,
  total_validators: 2, epoch: 101644, slot: 3252630
```

**この瞬間から、testnetcsmとtestnetcsm2が同じ鍵で同時に署名を開始しました。**

### 決定的な二重署名の瞬間

**2台が同じslotで同時に署名した記録：**

```
【testnetcsm2】
Jun 12 06:28:25 testnetcsm2 lighthouse[1101]:
  INFO  Successfully published attestations
  count: 1, validator_indices: [<validator_index_1>],
  slot: 3252692

【testnetcsm】
Jun 12 06:28:25 testnetcsm lighthouse[1220]:
  INFO  Successfully published attestations
  count: 1, validator_indices: [<validator_index_1>],
  slot: 3252692
```

**同じ時刻・同じvalidator（<validator_index_1>）・同じslot（3252692）で両方が署名を送信しました。これが二重署名です。**

### 重要な発見①：Lighthouseのログには警告が出ない

二重署名が発生しても、**Lighthouseのログには `Slashable` などの警告は出ませんでした。**

```
理由：
Lighthouseのスラッシング保護は
「自分自身の過去の署名DBと照合する」仕組み

testnetcsmのDB = 第1章の復元時点のスナップショット
testnetcsm2のDB = リアルタイム更新中

→ 2台はお互いの署名を知らない
→ 自分のDBと照合しても矛盾しない
→ 警告が出ないまま署名し続ける
```

**スラッシング保護DBは「同一マシン内での二重署名」を防ぐものです。別マシンが同時に署名することは防げません。**

### 重要な発見②：「即バン」ではない

二重署名から1時間以上経過しても、beaconcha.inのStatusは `Active` のままでした。

```
スラッシングの検知は即座ではない
  ↓
ネットワーク上のスラッシャーが
証拠（両方の署名データ）を収集する必要がある
  ↓
証拠が告発ブロックに組み込まれて初めて執行される
  ↓
テストネットではスラッシャーが少ないため
検知まで数時間かかることがある
```

### 重要な発見③：BeaconScoreが一時的に改善した

二重稼働によりアテステーション数が増加したため、BeaconScoreが一時的に改善しました。

| 時刻 | BeaconScore | 備考 |
|---|---|---|
| 14:11 JST（事前） | 23.93% | testnetcsm2のみ稼働 |
| 15:33 JST（Last 1h） | 92.16% | 両VM稼働・アテステーション増加 |
| 15:35 JST（Last 24h） | 27.82% | 24h全体で見ると低下中 |

**「スコアが上がった＝安全」ではありません。二重署名の証拠が集まり次第スラッシングが執行されます。**

### 重要な発見④：二重稼働でも報酬は1回分しか得られない

「2台が同時に署名すれば報酬が2倍になるのでは？」と思うかもしれませんが、そうはなりません。

```
同じslotに同じvalidatorから署名が2つ届いた場合：
  → ネットワークは1つだけ採用する
  → もう1つは「余分な署名」として無視される
  → 報酬は1回分のみ
```

| 項目 | 結果 |
|---|---|
| 報酬 | 変わらない（1回分のみ） |
| リスク | スラッシングが発生する |
| 結論 | **デメリットしかない** |

> 💡 **BeaconScoreが一時的に改善した理由：**
> testnetcsm2が署名できなかったslotを
> testnetcsmが補完したことで署名成功率が上がったためです。
> ただし報酬は1回分のみで、スラッシングリスクだけが増えた状態です。

### 重要な発見⑤：スラッシングに必要な二重署名は「1回」

スラッシングのしきい値は1回の二重署名です。何十回も繰り返す必要はありません。

```
条件：
  同じターゲットepochに対して
  異なる内容の署名が2つ存在する証拠が提出される
  → これだけでスラッシング確定

今回：
  slot 3252653で初回二重署名（06:20 UTC）
  slot 3252692で決定的二重署名（06:28 UTC）
  → 約2時間12分後にスラッシング検知
```

**回数ではなく「証拠がブロックに含まれた瞬間」に確定します。**

## 7. スラッシング検知後の観察（実録）

### CSM Sentinel 通知（実録）

> 💡 **Key indexとは：**
> Lido CSMに登録した鍵の順番を表す番号です。
> 最初に登録した鍵が `Key index: 0`、次が `Key index: 1` となります。
> 今回は2本の鍵を登録しているため、2通りの通知が順次届きました。
>
> | Key index | 対応する鍵 |
> |---|---|
> | Key index: 0 | `<validator_pubkey_1>`（`<validator_index_1>`） |
> | Key index: 1 | `<validator_pubkey_2>`（`<validator_index_2>`） |

**1本目（Key index: 0）：**
```
🚨 Validator slashing reported
  Validator: <validator_pubkey_1>
  Key index: 0
  nodeOperatorId: <your_node_operator_id>
```

**2本目（Key index: 1）：数分後に到着**
```
🚨 Validator slashing reported
  Validator: <validator_pubkey_2>
  Key index: 1
  nodeOperatorId: <your_node_operator_id>
```

> 💡 **2本の通知にタイムラグがありました。**
> スラッシングの証拠は鍵ごとに独立して収集されます。
> それぞれ別のブロックプロポーザーが証拠を提出したため
> 通知が順次届きました。

> 💡 **testnetcsmのサービスを停止した後も通知が届きました。**
> スラッシングの処理はネットワーク側で継続するため
> ノードを止めた後でも通知は届きます。
> Telegramの通知設定をオンにしておいてください。

### スラッシング通知後の初動対応

1本目のCSM Sentinel通知を受信した時点で
即座にtestnetcsmのサービスを停止しました。
2本目の通知はサービス停止後に届きました。

**なぜ速やかに停止すべきか：**

スラッシングのペナルティは2種類あり、対応が異なります。

| ペナルティ | タイミング | 停止の効果 |
|---|---|---|
| Initial Penalty（約1 ETH） | 証拠提出時に即座に確定 | **止めても変わらない** |
| Missed Attestation Penalty（約0.1〜0.2 ETH） | 36日間積み上がる | **速やかに停止することで最小化できる** |

```
【正確な理解】

速やかに停止しても
Initial Penalty（約1 ETH）は取り消せない
→ 証拠が提出された時点ですでに確定している

速やかに停止することで防げるのは
Missed Attestation Penalty の積み上がり
→ スラッシング後も署名を試みるが全てミス扱いになる
→ 停止することでこの追加損失を最小化できる
```

**通知が届いた時点で即座に停止するのが正解です。**

## 8. まとめ：今回の検証で得られた知見

| 知見 | 内容 |
|---|---|
| スラッシングは即座ではない | 二重署名後もしばらくActiveのまま。スラッシャーが証拠を集めてブロックに含めるまで時間がかかる |
| ログには警告が出ない | Lighthouseは別マシンの署名を知らない。ログに `Slashable` は出ない |
| 保護DBは万能ではない | スラッシング保護DBは「自分自身の過去の署名」との照合のみ。別マシンの同時署名は防げない |
| テストネットは検知が遅い | スラッシャーが少ないため、本番より検知に時間がかかる |
| BeaconScore一時改善 | 二重稼働でアテステーション数が増えスコアが上がる。スコア改善≠安全 |
| 二重署名の検知回数 | 1回の証拠提出でスラッシング確定。何度も繰り返す必要はない |
| 監視ツールが命綱 | CSM Sentinelなしでは検知が大幅に遅れる。構築時に必ず設定する |

### 本番環境で絶対に守るべきこと

```
【第1章との繋がり】

復元作業の最大リスク =「復元元と復元先の同時稼働」

今回の検証で体験したこと：
  2台が同時に稼働 → 二重署名 → スラッシング（執行済み）

守るべき手順：
  ① 旧環境のサービスを完全停止・確認
  ② 停止確認後に新環境を起動
  ③ 絶対に同時稼働させない
```

> 💡 **この手順は以下の全ての場面に適用されます：**
> - PC故障時の復元作業（第1章）
> - クライアントソフトの移行
> - ベアメタルへの引っ越し（第2部）
> - その他、同じ鍵を別環境で動かす全ての場面
>
> Lido公式ガイドでは「元の環境を停止後、
> 最低1時間待機してから新環境を起動する」ことを推奨しています。
> 1時間のオフラインはCSMの報酬に影響しません。
>
> 📎 [Lido CSM Slashing Prevention Guide](https://docs.lido.fi/staking-modules/csm/guides/slashing/)

### 監視ツールの重要性：リアルタイム検知のために

今回の検証で「スラッシングは即座に検知されない」ことが判明しました。
**だからこそ、リアルタイム監視ツールの設定が重要です。**

| ツール | 役割 | 設定タイミング |
|---|---|---|
| **CSM Sentinel**（Telegram Bot） | スラッシング・ペナルティをリアルタイム通知 | **バリデータ登録時に必ず設定** |
| **beaconcha.in アラート** | バリデータステータス変化をメール通知 | 構築完了時に設定 |
| **Doppelgänger Detection** | 起動時に同じ鍵の二重稼働を検知 | lighthouse-vc設定時に確認 |

> ⚠️ **CSM Sentinelは構築時に登録してください。**
> 今回の検証ではスラッシング発生後に登録しましたが
> 本来はバリデータ登録と同時に設定すべきです。
> 監視なき運用はリスクそのものです。

> 💡 **CSM Sentinelの登録方法・通知一覧は本章冒頭を参照してください。**

### Doppelgänger Detection について

今回のtestnetcsmのログに以下が記録されていました：

```
INFO  Doppelganger protection disabled.
```

**この機能が有効だった場合、二重稼働を早期に検知できた可能性があります。**

#### Doppelgänger Detection とは

ドイツ語で「二重に歩く者」を意味する言葉です。
Ethereumのコンテキストでは「別のマシンで動いている同じ鍵のバリデータ」を指します。

Lighthouse v1.5.0から搭載されたこの機能は、
lighthouse-vc起動時に**2〜3 epoch間サイレント状態を保ち**、
ネットワーク上に同じ鍵が存在するかどうかを監視してから署名を開始します。

```
【動作の流れ】

lighthouse-vc 起動
    ↓
2〜3 epoch間（約13〜19分）サイレント状態
  → ネットワーク上の同じ鍵の動きを監視
        ↓
同じ鍵の活動を検知した場合
  → 起動を中止する（スラッシング防止）

同じ鍵の活動を検知しなかった場合
  → 署名開始
```

#### 有効化方法

`lighthouse-vc.service` の `ExecStart` に以下のフラグを追加します：

```ini
ExecStart=/usr/local/bin/lighthouse vc \
  --network hoodi \
  ...既存のオプション...
  --enable-doppelganger-protection
```

#### 重要な注意点

| 項目 | 内容 |
|---|---|
| 完全性 | **完全ではない（ベストエフォート）** |
| 失敗ケース | ネットワーク障害等で検知できない場合がある |
| 位置づけ | 「最後の防衛線」であり、過信は禁物 |
| 副作用 | 起動後2〜3 epochは署名できない（軽微なペナルティ） |

> ⚠️ **Doppelgänger Detectionを有効にしていても、同じ鍵を2台で同時に動かすのは安全ではありません。** この機能はあくまでも保険です。二重起動自体を避けることが最重要です。

> 💡 **本番環境ではDoppelgänger Detectionを有効にしておくことを推奨します。**
>
> 📎 [Lighthouse公式ドキュメント：Doppelganger Protection](https://lighthouse-book.sigmaprime.io/validator-doppelganger.html)

> 💡 **筆者体験：**
> 検証後にソファで横になっていたところ、
> スマートフォンのTelegramにCSM Sentinelの通知が届き
> 即座に対応することができました。
> CSM Sentinelがなければ気づくまでにさらに時間がかかり
> その間もペナルティが積み上がっていたかもしれません。
> **24時間いつでも気づける監視体制が本番運用の生命線です。**

### スラッシング後の対応

スラッシングされた鍵は36日後（2026年7月19日）に強制退出し、残高が引き出し可能になります。

**Lido CSMへの手動報告は不要です。**
スラッシングの処理はスマートコントラクトが全て自動的に行います：

```
① スラッシング検知（自動）
② ボンドからInitial Penalty差し引き（自動）
③ 36日間のMissed Attestation Penalty（自動）
④ 強制退出・残高引き出し可能（自動）
```

> 💡 **新しい鍵の登録について**
> スラッシングされた鍵の退出後、
> 新しい鍵の生成・インポート・デポジット手順は
> 第2部・Phase 4（鍵の増設編）と同じ手順で実施できます。
