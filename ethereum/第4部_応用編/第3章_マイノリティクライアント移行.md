## 1. マイノリティクライアントとは

Ethereumネットワークには複数の実装（クライアント）が存在します。ノードオペレーターがどのクライアントを使うかは自由ですが、特定のクライアントに集中しすぎると**ネットワーク全体のリスクが高まります。**

### クライアント多様性の重要性

````
【危険なシナリオ】

あるクライアントが全ノードの66%以上を占めている場合：
  そのクライアントに重大なバグが発生
      ↓
  66%以上のノードが誤った判断をする
      ↓
  チェーンが不正なまま確定（ファイナライズ）
      ↓
  大規模なスラッシング発生
  ネットワーク停止の可能性
````

> 💡 **33%ルール：**
> Ethereum.orgは「どのクライアントも33%を超えるべきではない」と推奨しています。
> 33%を超えると単一障害点になりうるからです。

### 2026年現在のクライアントシェア

**実行レイヤー（EL）：**

| クライアント | シェア | 評価 |
|---|---|---|
| **Geth** | **41%** | 多すぎる・削減すべき |
| **Nethermind** | **38%** | 適切 |
| Besu | 16% | 少ない |
| Reth | 5.4% | 少ない |

**合意レイヤー（CL）：**

| クライアント | シェア | 評価 |
|---|---|---|
| **Lighthouse** | **52.65%** | 多すぎる・削減すべき |
| Prysm | 17.66% | やや多い |
| Teku | 13.36% | 適切 |
| **Nimbus** | **9.7%** | 少ない・使うべき |

### なぜNethermind + Nimbusを選ぶのか

````
Geth（41%）の代わりにNethermindを使う
  → 実行レイヤーの多様化に貢献

Lighthouse（52%）の代わりにNimbusを使う
  → 合意レイヤーの多様化に貢献

さらに：
  Nimbusは「軽量・省リソース」が特徴
  → ホームステーキングに最適
  → 低スペックPCでも運用可能
````

### コマンドの違い

**Geth vs Nethermind：ネットワーク指定**

```bash
# Geth：フラグで指定
ExecStart=/usr/bin/geth \
  --hoodi \          ← フラグ

# Nethermind：設定ファイル名で指定
ExecStart=/usr/local/bin/nethermind/nethermind \
  --config hoodi \   ← 設定ファイル名
```

**Lighthouse vs Nimbus：同期方法**

```bash
# Lighthouse：サービス起動時にCheckpoint Sync
ExecStart=/usr/local/bin/lighthouse beacon_node \
  --checkpoint-sync-url https://...   ← 起動オプション

# Nimbus：起動前に別コマンドで実行
sudo -u consensus /usr/local/bin/nimbus_beacon_node trustedNodeSync \
  --network=hoodi \
  --trusted-node-url=https://hoodi.beaconstate.ethstaker.cc \
  --data-dir=/var/lib/nimbus \
  --backfill=false
```

> 💡 **NimbusのCheckpoint Syncは驚異的な速さでした。**
> 今回の実測：**わずか約2分15秒で完了**
> （03:13:47 開始 → 03:16:02 完了）

## 4. Nethermind のインストール（実行レイヤー・EL）

### ユーザーとディレクトリの作成

```bash
# Nethermind専用ユーザーの作成
# なぜ専用ユーザーを作るのか：
# 万が一の侵害時にシステム全体への影響を最小化するため
sudo useradd --no-create-home --shell /bin/false nethermind
sudo mkdir -p /var/lib/nethermind
sudo chown -R nethermind:nethermind /var/lib/nethermind
```

### バイナリのダウンロードとインストール

```bash
# 最新バージョンを自動取得
RELEASE_URL="https://api.github.com/repos/NethermindEth/nethermind/releases/latest"
BINARIES_URL="$(curl -s $RELEASE_URL | jq -r '.assets[] | select(.name) | .browser_download_url' | grep linux-x64 | grep -v asc)"

echo "Downloading URL: $BINARIES_URL"

cd ~
wget -O nethermind.zip $BINARIES_URL
sudo mkdir -p /usr/local/bin/nethermind
sudo unzip nethermind.zip -d /usr/local/bin/nethermind
rm nethermind.zip

# バージョン確認
/usr/local/bin/nethermind/nethermind --version
```

**今回の実測：**
````
Version: 1.38.0+c07a4d65
Runtime: .NET 10.0.7
Platform: Linux x64
````

### JWT Secretの作成

```bash
# /secrets フォルダを作成
# なぜ/secrets か：
# システム全体で使う認証情報を一か所にまとめるため
sudo mkdir -p /secrets
openssl rand -hex 32 | tr -d "\n" | sudo tee /secrets/jwtsecret
sudo chmod 644 /secrets/jwtsecret
```

### systemdサービスの設定

```bash
sudo vi /etc/systemd/system/nethermind.service
```

```ini
[Unit]
Description=Nethermind Execution Layer Client service for Hoodi
Wants=network-online.target
After=network-online.target
Documentation=https://www.coincashew.com

[Service]
Type=simple
User=nethermind
Group=nethermind
Restart=always
RestartSec=3
KillSignal=SIGINT
TimeoutStopSec=900
WorkingDirectory=/var/lib/nethermind
Environment="DOTNET_BUNDLE_EXTRACT_BASE_DIR=/var/lib/nethermind"
ExecStart=/usr/local/bin/nethermind/nethermind \
  --config hoodi \
  --datadir="/var/lib/nethermind" \
  --Network.DiscoveryPort 30303 \
  --Network.P2PPort 30303 \
  --Network.MaxActivePeers 50 \
  --JsonRpc.Port 8545 \
  --JsonRpc.EnginePort 8551 \
  --Metrics.Enabled true \
  --Metrics.ExposePort 6060 \
  --JsonRpc.JwtSecretFile /secrets/jwtsecret

[Install]
WantedBy=multi-user.target
```

### 起動と確認

```bash
sudo systemctl daemon-reload
sudo systemctl enable nethermind
sudo systemctl start nethermind
sudo systemctl is-active nethermind
```

```bash
# ログをリアルタイムで確認する
# 「Initialization Completed」が表示されれば起動成功
sudo journalctl -fu nethermind
```

**起動直後の重要なログ：**

````
Initialization Completed
Chain ID: Hoodi
JSON RPC: http://127.0.0.1:8545 ; http://127.0.0.1:8551
Reading authentication secret from '/secrets/jwtsecret'
Waiting for Forkchoice message from Consensus Layer
````

> 💡 **「Waiting for Forkchoice message from Consensus Layer」について：**
> Post-Merge以降のEthereumでは、EL（Nethermind）はCLからの
> Forkchoiceメッセージが届くまで同期を開始できません。
> これはGethも同様の動作です。
> 第1部・第2部でもGeth起動後にLighthouseを起動するまで
> 同じ状態が続いていました。
> CLクライアント（Nimbus）を起動することで同期が始まります。
> これは正常な動作です。

## 5. Nimbus のインストール（合意レイヤー・CL / Beacon Node）

### ユーザーとディレクトリの作成

```bash
# consensus専用ユーザーの作成
sudo adduser --system --no-create-home --group consensus
sudo mkdir -p /var/lib/nimbus
sudo chown -R consensus:consensus /var/lib/nimbus

# BN用の空ディレクトリを作成（重要）
# なぜ別ディレクトリが必要か：
# Nimbus BNは --in-process-validators=false を指定しても
# 同じ --data-dir のvalidatorsディレクトリのkeystoreをロックしてしまう
# BNとVCが同じvalidatorsディレクトリを共有するとVCが起動できなくなる
sudo mkdir -p /var/lib/nimbus-bn/validators
sudo mkdir -p /var/lib/nimbus-bn/secrets
sudo chown -R consensus:consensus /var/lib/nimbus-bn

# 依存パッケージのインストール
sudo apt install libsnappy-dev libc6-dev libc6 ccze -y
```

> ⚠️ **`/var/lib/nimbus-bn/` は必ず作成してください。**
> この手順を省略するとNimbus VCが起動できなくなります。
> 詳細はセクション7のトラブルシューティングを参照してください。

### バイナリのダウンロードとインストール

```bash
RELEASE_URL="https://api.github.com/repos/status-im/nimbus-eth2/releases/latest"
BINARIES_URL="$(curl -s $RELEASE_URL | jq -r '.assets[] | select(.name) | .browser_download_url' | grep _Linux_amd64 | grep -v asc)"

echo "Downloading URL: $BINARIES_URL"

cd ~
wget -O nimbus.tar.gz $BINARIES_URL
tar -xzvf nimbus.tar.gz -C ~
mv ~/nimbus-eth2_Linux_amd64_* ~/nimbus
sudo mv ~/nimbus/build/nimbus_beacon_node /usr/local/bin
sudo mv ~/nimbus/build/nimbus_validator_client /usr/local/bin
rm nimbus.tar.gz
rm -rf ~/nimbus

# バージョン確認
nimbus_beacon_node --version
```

**今回の実測：**
````
Nimbus beacon node v26.5.0-6fb05f-stateofus
Ethereum consensus spec v1.7.0-alpha.7
````

### Checkpoint Syncで高速同期

```bash
# 起動前に信頼できるノードから最新状態を取得する
# なぜCheckpoint Syncを使うのか：
# 通常の同期は数時間〜数日かかるが
# Checkpoint Syncを使えば数分で完了する
sudo -u consensus /usr/local/bin/nimbus_beacon_node trustedNodeSync \
  --network=hoodi \
  --trusted-node-url=https://hoodi.beaconstate.ethstaker.cc \
  --data-dir=/var/lib/nimbus \
  --backfill=false
```

**完了メッセージ：**
````
NTC Done, your beacon node is ready to serve you!
````

> 💡 **Checkpoint Syncの実測時間：約2分15秒**
> 03:13:47 開始 → 03:16:02 完了
> Lighthouseの場合と同様に数分で完了します。

### systemdサービスの設定

```bash
sudo vi /etc/systemd/system/nimbus.service
```

```ini
[Unit]
Description=Nimbus Consensus Layer Client service for Hoodi
Wants=network-online.target
After=network-online.target
Documentation=https://www.coincashew.com

[Service]
Type=simple
User=consensus
Group=consensus
Restart=on-failure
RestartSec=3
KillSignal=SIGINT
TimeoutStopSec=900
ExecStart=/usr/local/bin/nimbus_beacon_node \
  --network=hoodi \
  --data-dir=/var/lib/nimbus \
  --validators-dir=/var/lib/nimbus-bn/validators \
  --secrets-dir=/var/lib/nimbus-bn/secrets \
  --tcp-port=9000 \
  --udp-port=9000 \
  --max-peers=100 \
  --rest-port=5052 \
  --enr-auto-update=true \
  --non-interactive \
  --status-bar=false \
  --in-process-validators=false \
  --web3-url=http://127.0.0.1:8551 \
  --rest \
  --metrics \
  --metrics-port=8008 \
  --jwt-secret="/secrets/jwtsecret" \
  --suggested-fee-recipient=<Lido_CSM_Fee_Recipient_Address>

[Install]
WantedBy=multi-user.target
```

> 💡 **`--validators-dir` と `--secrets-dir` について：**
> BN用の空ディレクトリを指定します。
> これによりBNはVC用のkeystoreをロックしなくなります。
> `--in-process-validators=false` だけでは不十分で
> この設定が必須です。

> 💡 **`--suggested-fee-recipient` について：**
> Nimbusでは**BN側**でfee-recipientを設定します。
> Lighthouseでは**VC側**で設定していた点が異なります。
>
> | ネットワーク | Lido CSM Fee Recipient Address |
> |---|---|
> | Hoodi Testnet | `0x9b108015fe433F173696Af3Aa0CF7CDb3E104258` |
> | Mainnet | `0x388C818CA8B9251b393131C08a736A67ccB19297` |

> 💡 **`--in-process-validators=false` について：**
> BNとVCを分離して動かすための設定です。
> Nimbusは本来BNとVCを一体化して動かすこともできますが
> 第1部・第2部と同様にBN/VC分離構成を採用します。

> 💡 **`doppelgangerDetection: true`（デフォルト有効）**
> Nimbusは**デフォルトでDoppelgänger Detectionが有効**です。
> 第2章で検証したtestnetcsmでは無効になっていましたが
> Nimbusでは何もしなくても有効になっています。

## 6. リソース比較（実測値）

### 同期時間比較

| 項目 | Geth（第1部・ゼロから） | Nethermind（今回・ゼロから） |
|---|---|---|
| **合計時間** | 約5〜6時間 | **約1時間44分** ✅ |
| **段階数** | 2段階 | 3段階 |
| **chain/Headers** | 約1時間 | 約21分 |
| **state/Phase 1** | 約4〜5時間 | 約1時間10分 |
| **追加フェーズ** | なし | Large storage 約13分 |

> 💡 **Nethermindの同期時間の実測値：**
> 03:00 UTC 起動 → 04:44 UTC Snap sync completed
> **約1時間44分でGethの約1/3の時間で完了しました。**

### Nethermindの同期3段階

Gethは2段階（chain → state）ですが、Nethermindは3段階に分かれています。

```
【Stage 1】Old Headers（約21分）
  ブロックヘッダーのダウンロード
  ログ例：Old Headers 1,409,856 / 3,006,101 (46.90%)

【Stage 2】Snap State Ranges Phase 1（約1時間10分）
  通常アカウント・コントラクトのState同期
  ログ例：Snap State Ranges (Phase 1): (60.16%)

【Stage 3】Large storage（約13分）
  大容量コントラクト（DeFi等）のストレージ同期
  ログ例：Snap Large storage left: 1 (62.40%)

完了メッセージ：
  Snap sync completed.
```

> 💡 **「Large storage」とは：**
> UniswapやDAIなどのDeFiコントラクトが持つ
> 巨大なストレージデータです。
> 通常アカウントと分離して処理することで
> より効率的な同期を実現しています。

> 💡 **同期完了後もOld Bodiesの補完が続きます：**
> `eth_syncing`が`false`になる前に
> `Old Bodies: X / 3,006,100 (X%)`のログが出ます。
> これはNimbusのbackfillと同様に
> 過去データの補完処理であり
> バリデータの動作には影響しません。

### 鍵の生成

> 💡 **validator_start_indexについて：**
> 鍵の番号はVMごとではなく**ニーモニックグループ全体**で管理します。
>
> | index | 鍵 | 用途 |
> |---|---|---|
> | 0 | 鍵A | testnetcsm2（スラッシング済み） |
> | 1 | 鍵B | testnetcsm2（スラッシング済み） |
> | 2 | 鍵C | testnetcsm2（Depositable） |
> | **3** | **鍵D** | **testnetcsm（今回）** |
>
> index 0〜2はすでに使用済みのため
> testnetcsmでは `--validator_start_index 3` を指定します。

```bash
# csm-artifactsの確認
ls ~/csm-artifacts/ 2>/dev/null || echo "csm-artifactsが存在しません"

# 存在しない場合は以下でダウンロード

cd ~/csm-artifacts
RELEASE_URL=$(curl -s https://api.github.com/repos/ethstaker/ethstaker-deposit-cli/releases/latest \
  | jq -r '.assets[] | select(.name | contains("linux-amd64") and (contains("sha256")|not)) | .browser_download_url')
wget $RELEASE_URL -O ethstaker_deposit-cli-linux-amd64.tar.gz
tar -xvf ethstaker_deposit-cli-linux-amd64.tar.gz
```

```bash
# 既存ファイルの確認（古いファイルが残っていないか）
ls ~/csm-artifacts/ethstaker_deposit-cli-*/validator_keys/ 2>/dev/null

# 鍵の生成（index 3から）

  --num_validators 1 \
```

### systemdサービスの設定

```bash
sudo vi /etc/systemd/system/nimbus-vc.service
```

```ini
[Unit]
Description=Nimbus Validator Client service for Hoodi
Wants=network-online.target
After=network-online.target nimbus.service
Documentation=https://www.coincashew.com

[Service]
Type=simple
User=consensus
Group=consensus
Restart=on-failure
RestartSec=30
KillSignal=SIGINT
TimeoutStopSec=900
ExecStart=/usr/local/bin/nimbus_validator_client \
  --data-dir=/var/lib/nimbus \
  --beacon-node=http://127.0.0.1:5052 \
  --metrics \
  --metrics-port=8009 \
  --suggested-fee-recipient=<Lido_CSM_Fee_Recipient_Address>

[Install]
WantedBy=multi-user.target
```

> 💡 **`RestartSec=30` について：**
> 起動失敗時の再起動間隔を30秒にします。
> 3秒のままにするとエラー時に大量のプロセスが
> 生成されVMがフリーズする原因になります。

> ⚠️ **`--keymanager`オプションについて：**
> `--keymanager`を使う場合は
> `--keymanager-token-file`の指定が必須です。
> 今回はKeymanager APIは不要なため省略しています。

> 💡 **末尾の`\`に注意：**
> 最終行のオプションに`\`を付けると起動エラーになります。
> 最終行には`\`を付けないでください。

### トラブルシューティング：VMフリーズとkeystoreロックエラー

**症状：**
```
ERR Unable to lock keystore file
  error_msg="Resource temporarily unavailable"
FAT Unable to load keystore
```

**原因：**
```
Nimbus BNが --in-process-validators=false を指定しても
同じ --data-dir のvalidatorsディレクトリの
keystoreをロックしてしまう

→ nimbus-vcが起動できず3秒ごとに再起動
→ 大量プロセスがVMのリソースを消費
→ VMがフリーズ
```

**解決手順：**

```bash
# 1. 両サービスを停止
sudo systemctl stop nimbus-vc
sudo systemctl stop nimbus

# 2. BN用の空ディレクトリを作成
sudo mkdir -p /var/lib/nimbus-bn/validators
sudo mkdir -p /var/lib/nimbus-bn/secrets
sudo chown -R consensus:consensus /var/lib/nimbus-bn

# 3. nimbus.serviceに以下のオプションを追加
sudo vi /etc/systemd/system/nimbus.service
```

追加するオプション（`--data-dir`の直後）：
```ini
  --validators-dir=/var/lib/nimbus-bn/validators \
  --secrets-dir=/var/lib/nimbus-bn/secrets \
```

```bash
# 4. nimbus-vc.serviceのRestartSecを変更
sudo vi /etc/systemd/system/nimbus-vc.service
# RestartSec=3 → RestartSec=30 に変更

# 5. 設定を反映してBNを先に起動
sudo systemctl daemon-reload
sudo systemctl start nimbus
sleep 20

# 6. BNがactiveになってからVCを起動
sudo systemctl is-active nimbus
sudo systemctl start nimbus-vc
sudo systemctl is-active nimbus nimbus-vc
```

> ⚠️ **BNを先に起動してから20秒待ってVCを起動してください。**
> 起動順序を守らないとVCが先に起動してエラーになります。

**確認できた正常なディレクトリ構成：**

```
/var/lib/nimbus/          ← VCが使うメインディレクトリ
  db/                     ← BNのチェーンデータ
  secrets/                ← VCのパスワードファイル
  validators/             ← VCの鍵データ・slashing_protection DB

/var/lib/nimbus-bn/       ← BN用の空ディレクトリ
  secrets/                ← 空
  validators/             ← 空（BNはここを参照する）
```
