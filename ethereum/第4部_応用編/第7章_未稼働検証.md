## 1. 検証の目的

第3章・第5章では、クライアントの移行や構成変更を扱いました。本章では
視点を変え、「バリデータを意図的に停止し、放置するとどうなるか」を
実際に確かめます。単純なペナルティ額の確認だけでなく、運用者として
以下の点を明らかにすることが目的です。

```
① ペナルティの大きさ・蓄積ペースはどの程度か
② 放置すると監視・アラート（CSM Sentinel）は気づいてくれるのか
③ 放置し続けると強制退場（Exit/Ejection）になるのか
④ Lido CSM独自の「Strikesシステム」は、実際何回・いつ付与されるか
⑤ Strikesが閾値に達した場合、実際にForce Ejectionされるか
⑥ Force Ejectionが発生した場合、CSM Sentinelは通知してくれるか
```

①②③は本章執筆時点で確定していますが、④⑤⑥は観察継続中です。

## 5. 確定している結論

### ② CSM Sentinel（Telegram通知）は気づいてくれるか

**結論：気づかない（今回の規模・期間では無反応）。**

未稼働検証の開始から本章執筆時点まで、Sentinelからの通知は一切来ていない。
BeaconScoreが90%台から10%台後半まで落ちる規模の劣化が起きても、Sentinelは
沈黙していた。

> **実用的な学び：** CSM Sentinelの監視対象は、スラッシングやExitのような
> 「イベント系」であり、Inactivity Leakのような地味な「状態系の劣化」は
> 検知してくれない可能性が高い。「Sentinelが鳴らないから大丈夫」と判断
> するのは早計。別の監視レイヤーが必要（詳細はセクション7）。

### ③ 放置し続けると強制退場（Exit/Ejection）になるか

**結論：レイヤーによって答えが異なる。Ethereumプロトコル本体は「ならない」、
Lido CSM独自レイヤーは「ある条件下でなる」（詳細・継続観察はセクション8）。**

```
Ethereumプロトコル本体：
  - Inactivity Leakは報酬が削られるだけのペナルティで、強制退場の
    仕組みではない
  - 唯一の強制退場ルートはバランスが16 ETH閾値を割り込んだ場合のみ
  - 今回のペナルティペース（数日で0.001〜0.005 ETH程度）では、
    16 ETH減るまでに天文学的な時間がかかる
```

## 6. サーバーが物理的に消失した場合はどうなるか

検証中に浮かんだ追加の問い：「VMを止める」のではなく「サーバー自体が
物理的になくなった」場合、プロトコルから見て挙動は変わるのか。

**結論：プロトコルレベルでは完全に同一視される。**

Ethereumのコンセンサス層が観測できる状態は`Pending → Active → Exited → Withdrawn`
の4つのみ。「サーバーが壊れた」「ソフトウェアを止めた」という"理由"は
プロトコルには伝わらず、「Attestationが来ない」という事実だけが観測される。

```
サーバー消失後の挙動（プロトコル単体）：
① Activeのまま残り続ける（永久に）
② Inactivity Leakで報酬が削られ続ける
③ CSM Sentinelはこの状態に気づかない（上記②の通り）
④ 16 ETH閾値に達するまで強制Exitされない（数年〜数十年オーダー）
⑤ ただし、Lido CSM独自のStrikesシステムが別途働く可能性がある（セクション8）
```

> **実用的な学び：** もしサーバーが物理的に失われ、誰も気づかなければ、
> Ethereumプロトコル単体では「生きているように見える」状態が続く。
> 実際にいつ・どう退場処理されるかは、CSM独自のStrikesシステムに
> 依存する。死活監視（セクション7）の必要性を裏付ける根拠になる。

## 7. 死活監視・アラート体制の課題

今回の検証で「CSM Sentinelだけでは不十分」という実用上の発見があった。
イベント系（スラッシング・Exit）は拾えるが、地味な性能劣化（Inactivity
Leak等）には無反応なため、別の監視レイヤーが必要になる。

### 候補①：Grafana Alerting（既存資産の延長・優先度高）

```
- 第2部で構築済みのGrafanaダッシュボードの延長で導入できる
- BeaconScore（30d）が一定閾値（例：95%）を下回ったらアラート
- Telegram/Discord/メールへのWebhook通知も無料で設定可能
```

### 候補②：Uptime Kuma（外部監視・無料・自己ホスト）

```
- オープンソースの死活監視ツール
- HTTPエンドポイントやAPIレスポンスを定期チェックして外部から死活監視
```

### 候補③：自作Cron + beaconcha.in API + Telegram Bot

```
- node_check.sh の延長線上で作りやすい
- beaconcha.in APIでBeaconScore・バランスの変化を定期取得
- 閾値を下回ったらTelegram Botで通知
```

> 今後の宿題として記録。着手は別セッションで予定。

## 8. 【発見】Lido CSM独自の「Strikesシステム」

### 8-1. 発見の経緯

別の検証（Google Cloud無料枠で構築したバリデータ）で、以下のような
CSM Sentinel通知ログが残っていた。

```
🚨 Strikes penalty processed
  Penalty amount: 0.258 ether
  Penalty will be charged when the validator withdraws

🚨 Exit fee recorded
  Fee paid now: 2 wei / Fee to be charged on exit: 2 wei

👀 Information about validator withdrawal has been submitted
  Exit balance: 31.96988154 ether

📈 Rewards distributed!
  ⚠️ Strikes detected for your validators
```

ペナルティ確定（最初の通知）から退場提出（3番目の通知）まで、わずか
9分で完了していた。これは本章で観察しているInactivity Leak（じわじわ
削れるだけで何も起きない）とは全く異なる、即時的な強制退場の実例。

### 8-2. 公式ドキュメントによる裏付け

Lido公式ドキュメント（Penalties・Join CSM）によると、CSM v2で導入された
Strikesシステムは、Ethereumプロトコル本体とは別の、独自の強制退場
メカニズムです。

```
- ストライクとは、評価フレーム単位でバリデータのパフォーマンスが
  しきい値を下回ったことを示す記録
- ストライク数が閾値（公式ドキュメントの例：6か月で3回）に達すると、
  permissionless（誰でも実行可能）な方法でそのバリデータの退場を
  トリガーできる
- 退場時、badPerformancePenalty（DAOが設定する固定額）がNode
  Operatorのボンドから没収される
- ストライクは一定期間（ライフタイム）を超えると古いものから失効する
```

### 8-3. EthereumプロトコルとCSM独自レイヤーの違い

| レイヤー | 強制退場の有無 | トリガー条件 |
|---|---|---|
| Ethereumプロトコル本体 | なし | バランスが16 ETH閾値を割り込んだ場合のみ（現実的には到達しない） |
| **Lido CSM（独自レイヤー）** | **あり** | **Strikesが一定数累積した場合** |

> これは本章セクション5・③の結論を補足する重要な点。「強制退場には
> ならない」という結論は、Ethereumプロトコル本体のみを見た場合の話で、
> **CSM独自のStrikesシステムを踏まえると話が変わる。**

### 8-4. 継続観察の方針と確認方法

本章執筆時点で約3日が経過しているが、Strikesの評価サイクル（フレーム
単位、おそらく日〜週オーダー）にまだ到達していない可能性が高い。

**【発見】CSM UI（csm.testnet.fi）に「With strikes」専用カウンターがある**

CSM公式UIのDashboard画面、Keys Breakdownセクションに、Strikesが付与
された鍵の数を直接示すカウンターが存在することを確認した。

```
Keys Breakdown：
  Active              : 2
  Exited              : 2
  Withdrawn           : 1
  With strikes        : 0   ← このカウンターが0から1以上に変化した
                              タイミングで、Strikes発生を検知できる
```

これはbeaconcha.inでの個別確認（BeaconScore・Atts等）よりも直接的で
効率的な確認方法。今後の定期観察は、このカウンターの確認を中心に行う。

**観察したいポイント：**

```
- Strikesが何回・いつ付与されるか
- 何回目のStrikesでForce Ejectionがトリガーされるか
  （公式ドキュメント上の例は3回だが、実際のパラメータは要確認）
- Force Ejection時、CSM Sentinelは通知してくれるか
  （8-1の実例では通知が来ていたため、Strikes関連は監視対象に
    入っている可能性が高い）
- badPerformancePenaltyとして実際にいくら没収されるか
```

> 💡 **本章は、上記の観察結果が得られた時点で追記・更新する予定です。**
> 現時点（With strikes: 0）では、Strikes発生・Force Ejectionの実例は
> まだ確認できていません。

## 9. 学んだこと・教訓

- **死活監視には「イベント系」と「状態系」の2種類があり、片方しか
  カバーしていないと盲点が生まれる：** CSM Sentinelはスラッシング・
  Exitのようなイベントには気づくが、Inactivity Leakのような地味な
  状態劣化には無反応だった。
- **「強制退場の有無」はレイヤーによって答えが異なる：** Ethereum
  プロトコル本体の話とLido CSM独自レイヤーの話を混同すると、誤った
  結論に至る。
- **検証対象の全バリデータについて、開始前の基準値を必ず記録する：**
  比較対象（今回はtestnetcsm2）のBefore値を記録し忘れたことで、
  構成差の原因分析が「推測の域」に留まってしまった。
- **公式UIには、見落としがちな便利な確認手段が用意されている場合
  がある：** CSM Dashboardの「With strikes」カウンターは、地道な
  個別確認より効率的な定点観測ポイントだった。
