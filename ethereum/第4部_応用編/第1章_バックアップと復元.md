## 1. なぜバックアップが必要か

バリデータ運用において「PCが突然壊れる」リスクは常に存在します。

Ethereumネットワークのデータ（約121GB）は再ダウンロードできますが、**あなたの鍵データは二度と再現できません**。バリデータ鍵を失うと、デポジットした資金を引き出すことができなくなります。

本章では以下の2つを目的として記録を残します。

1. **何をバックアップすべきか**を正確に理解する
2. **実際に別VMへ復元できるか**を検証し、手順を確定する

「いざとなれば復元できる」という根拠のない自信ではなく、**実際に復元してみた記録**をここに残します。

## 2. 守るべきデータの正体（実測値）

### 各ディレクトリのサイズと重要度

> 参考実測値（Hoodi Testnet / バリデータ2個 / VM環境 / 2026年6月時点）

| ディレクトリ | サイズ | バックアップ要否 | 理由 |
|---|---|---|---|
| `/var/lib/ethereum/geth` | 93GB | **不要** | チェックポイント同期で再構築可能 |
| `/var/lib/ethereum/lighthouse` | 28GB | **不要** | 同上 |
| `/var/lib/lido-csm` | 21MB | **必須（命のデータ）** | バリデータ鍵・スラッシング保護DB |

### 「121GBはいらない。守るべきデータはたった21MBだ」

この事実は、初めて知ったとき驚かされます。

**Gethのデータ（93GB）とLighthouseのデータ（28GB）は、復元後にチェックポイント同期で再構築できます。** チェックポイント同期については第1部・Step 5で解説した通り、Ethereum Foundationが運営する信頼できるサーバーから最新の確定済みブロックをダウンロードする仕組みです。

一方、`/var/lib/lido-csm/` にはあなただけが持つ固有のデータが入っています。

```
/var/lib/lido-csm/
├── validators/
│   ├── validator_definitions.yml   ← バリデータ鍵の定義ファイル
│   ├── voting-keystore-*.json      ← 暗号化されたバリデータ鍵（実体）
│   └── slashing_protection.sqlite  ← スラッシング保護データベース
└── api-token.txt                   ← Lighthouse VC の認証トークン
```

この21MBを失うと：
- バリデータ鍵を再インポートできない
- スラッシング保護DBがなければ安全に再起動できない
- 最悪の場合、スラッシング（重大ペナルティ）のリスクを負いながら復元作業をすることになる

**バックアップとは「121GBのチェーンデータ」ではなく「21MBの鍵と保護データ」を守ることです。**

## 3. node_backupスクリプトの設定

### バックアップ対象ファイル一覧

`node_backup` スクリプトは以下のファイル・ディレクトリをまとめてアーカイブします。

| 対象 | 内容 |
|---|---|
| `/var/lib/lido-csm/` | バリデータ鍵・スラッシング保護DB・APIトークン（最重要） |
| systemdサービスファイル4種 | geth / lighthouse / lighthouse-vc / mev-boost の設定 |
| `node_check.sh` | 日常運用の状態確認スクリプト |
| `node_safe_stop.sh` | 安全停止スクリプト |
| `/etc/chrony/chrony.conf` | 時刻同期設定（アテステーション精度に影響） |

systemdサービスファイルをバックアップに含める理由は、**復元先で `systemctl daemon-reload` → `systemctl start` を即座に実行できるようにするためです。** 設定ファイルが手元になければ、サービスをゼロから書き直す必要があります。

### スクリプトの配置手順

> ⚠️ **このスクリプトはVM検証環境用のシンプルな構成です。**
> ベアメタル本番環境（第2部）では、鍵ファイルを除外する安全設計の
> スクリプトを使用しています。詳細は第2部・Phase 7を参照してください。
> 本章ではVM環境での動作検証を目的として、以下のシンプル版を使用します。

```bash
# スクリプトを作成する
# sudo = システムディレクトリ（/usr/local/bin/）への書き込みに管理者権限が必要
sudo vi /usr/local/bin/node_backup
```

> viエディタの基本操作：
> - `i` : 入力モードを開始する（文字が打てるようになる）
> - `Esc` : 入力モードを終了する
> - `:wq` : 保存して終了（write + quit）
> - `:q!` : 保存せずに強制終了
>
> viが開いたら `i` を押して入力モードにし、以下のスクリプト全体をコピー＆ペーストしてください。
> 貼り付け後は `Esc` → `:wq` で保存します。

```bash
#!/bin/bash
# --- 設定 ---
BACKUP_DIR="/home/<your_user>/node_backups"
TIMESTAMP=$(date +"%Y%m%d_%H%M%S")
BACKUP_FILE="${BACKUP_DIR}/lido_csm_backup_${TIMESTAMP}.tar.gz"
MAX_BACKUPS=7 # 過去7回分を保持

echo "Starting Ethereum Node Backup: $TIMESTAMP"

# 1. 命のデータを圧縮
# /var/lib/lido-csm/ を丸ごと固めることで、鍵もスラッシングDBも一網打尽にする
tar -czpf "$BACKUP_FILE" \
    /var/lib/lido-csm/ \
    /etc/systemd/system/geth.service \
    /etc/systemd/system/lighthouse.service \
    /etc/systemd/system/lighthouse-vc.service \
    /etc/systemd/system/mev-boost.service \
    /home/<your_user>/node_check.sh \
    /home/<your_user>/node_safe_stop.sh \
    /etc/chrony/chrony.conf 2>/dev/null

# 2. バックアップファイルの所有者を一般ユーザーに変更
chown <your_user>:<your_user> "$BACKUP_FILE"
chmod 600 "$BACKUP_FILE"

# 3. 古いバックアップの削除（MAX_BACKUPS を超えた分を消す）
ls -t ${BACKUP_DIR}/lido_csm_backup_*.tar.gz | tail -n +$((MAX_BACKUPS + 1)) | xargs -r rm 2>/dev/null

echo -e "Backup completed successfully! -> \033[0;32m$BACKUP_FILE\033[0m"
```

> 💡 **`<your_user>` はご自身のユーザー名に置き換えてください。**
> 今回の検証環境では `test` です。
> 置き換え箇所は3か所（`BACKUP_DIR`・`chown`・`node_check.sh`パス）あります。

```bash
# 実行権限を付与する
# chmod +x = 「このファイルをプログラムとして実行できる」属性を付ける
# これがないと「Permission denied」エラーになる
sudo chmod +x /usr/local/bin/node_backup
```

```bash
# バックアップファイルの保存先ディレクトリを作成する
# -p オプション = 途中のディレクトリも含めて一括作成（既に存在してもエラーにならない）
mkdir -p ~/node_backups
```

### バックアップの実行

```bash
# バックアップを実行する
# sudo = /var/lib/lido-csm/ は ethereum ユーザー所有のため、
#        root権限でないと読み込めないファイルが含まれる
sudo node_backup
```

**実行結果の例：**

```
Starting Ethereum Node Backup: 20260610_224019
Backup completed successfully! -> /home/test/node_backups/lido_csm_backup_20260610_224019.tar.gz
```

【読み方】
- `20260610_224019` : タイムスタンプ（2026年6月10日 22:40:19）。複数バックアップを区別するために使います。
- `Backup completed successfully!` : このメッセージが出ればバックアップ成功です。エラーが出た場合は権限不足の可能性があります。

### バックアップ内容の確認

```bash
# アーカイブの中身を確認する（展開はしない）
# -t : 内容を一覧表示（t = table of contents）
# -z : gzip圧縮ファイルを処理する
# -f : 対象ファイルを指定する
# <TIMESTAMP> は実際のタイムスタンプに置き換えてください
tar -tzf ~/node_backups/lido_csm_backup_<TIMESTAMP>.tar.gz
```

`var/lib/lido-csm/` 配下のファイルがリストに含まれていることを確認してください。

> 💡 **バックアップファイルのサイズについて**
>
> 21MBのデータを `tar.gz` で圧縮すると、実際のファイルサイズは**約1.2MB**になります。
> USBメモリや外付けSSD、クラウドストレージのどこにでも保存できるサイズです。
> **バックアップファイルはオフラインのストレージにも必ずコピーしてください。**
> クラウドストレージのみに保存すると、アカウントロック時にアクセスできなくなります。

## 4. 復元手順

### 前提条件

> ⚠️ **復元作業を開始する前に、必ずバックアップ元のサービスをすべて停止してください。**
>
> 同じバリデータ鍵を持つノードが**2台同時に稼働するとスラッシング（重大ペナルティ）が発生します。**
>
> スラッシングとは、二重署名（同じスロットに複数の矛盾した署名を提出すること）に対するペナルティで、デポジットしたETHが没収されます。Lidoのボンドも削られます。
>
> 今回の検証でも、「復元先VMを起動したまま復元元VMのサービスを停止し忘れる」ケースに危うく陥るところでした。**「バックアップ元を停止してから復元先を起動する」この順序を絶対に守ってください。**

バックアップ元のサービス停止手順：

```bash
# バックアップ元VMでサービスを停止する
# 停止順序：VC → Lighthouse → Geth（起動の逆順）
# なぜ逆順か：VCが署名作業を終えてから下位レイヤーを停止させるため
sudo systemctl stop lighthouse-vc
sudo systemctl stop lighthouse
sudo systemctl stop geth

# 停止確認（全て inactive になっていることを確認する）
sudo systemctl status geth lighthouse lighthouse-vc
```

### VirtualBox環境での転送方法

VM間でファイルを転送する方法はいくつかありますが、**NATネットワーク接続ではVM同士が直接通信できません。** VirtualBoxのNAT設定では各VMが独立したネットワーク空間に置かれるため、VM-AからVM-BのIPアドレスへ直接 `scp` することができません。

今回はホストPCの「共有フォルダ」を中継点として使用します。

```
【転送の流れ】

転送元VM ──┐
           ├── ホストPCの共有フォルダ（中継点） ──── 復元先VM
復元先VM ──┘
```

**手順1：VirtualBoxで共有フォルダを設定する**

VirtualBoxマネージャーで転送元VM → 設定 → 共有フォルダ → 追加：
- フォルダーパス：ホストPCの任意のフォルダ（例：`C:\backup_transfer`）
- フォルダー名：`backup_transfer`
- 自動マウント：チェックしない（手動でマウントする）

**手順2：転送元VMでマウントしてバックアップファイルをコピーする**

```bash
# マウントポイントを作成する（共有フォルダの「接続口」を作る）
sudo mkdir -p /mnt/backup_transfer

# 共有フォルダをマウントする
# -t vboxsf : VirtualBox共有フォルダ（vboxsf）形式として接続する
# backup_transfer : VirtualBox設定で付けたフォルダー名と一致させること
sudo mount -t vboxsf backup_transfer /mnt/backup_transfer

# バックアップファイルを共有フォルダにコピーする
# <TIMESTAMP> は実際のタイムスタンプに置き換えてください
sudo cp ~/node_backups/lido_csm_backup_<TIMESTAMP>.tar.gz /mnt/backup_transfer/
```

**手順3：復元先VMにも同じ共有フォルダを設定してマウントする**

VirtualBoxマネージャーで復元先VM → 設定 → 共有フォルダ → 同じフォルダを追加します（フォルダー名 `backup_transfer` で統一）。

```bash
# 復元先VMでもマウントポイントを作成してマウントする
sudo mkdir -p /mnt/backup_transfer
sudo mount -t vboxsf backup_transfer /mnt/backup_transfer
```

**手順4：バックアップファイルをホームディレクトリにコピーする**

```bash
# 共有フォルダからホームディレクトリにコピーする
# /var/lib/ への直接展開は権限エラーになるため、
# 一旦ホームディレクトリ（~/）にコピーしてから展開する
cp /mnt/backup_transfer/lido_csm_backup_<TIMESTAMP>.tar.gz ~/
```

### 復元の実行

```bash
# 既存データを念のためバックアップしておく（展開失敗時の保険）
# mv = ディレクトリ名を変えるだけなので、データは削除されない
sudo mv /var/lib/lido-csm /var/lib/lido-csm.bak

# アーカイブを展開して復元する
# -x : 展開（extract）
# -z : gzip圧縮を解凍する
# -p : 元のパーミッション（権限設定）を保持する（これが重要）
# -f : 対象ファイルを指定する
# -C / : ルートディレクトリ（/）を起点として展開する
#         アーカイブ内のパス「var/lib/lido-csm/...」がそのまま再現される
sudo tar -xzpf ~/lido_csm_backup_<TIMESTAMP>.tar.gz -C /
```

> 💡 **`-C /` オプションについて**
>
> `-C /`（大文字のC）は「このディレクトリを基準に展開する」という指定です。
> アーカイブ内のファイルパスが `var/lib/lido-csm/validators/...` のように
> ルートからの相対パスで記録されているため、`-C /` を指定することで
> `/var/lib/lido-csm/validators/...` に正しく展開されます。

```bash
# 展開結果を確認する
# -la : 隠しファイルも含め、権限・所有者・更新日時を詳細表示する
sudo ls -la /var/lib/lido-csm/
```

`ethereum` ユーザーが所有者になっており、`validators/` ディレクトリと `api-token.txt` が存在することを確認してください。

### サービスの起動

サービスは必ず **Geth → Lighthouse → Lighthouse-VC** の順で起動します。

なぜこの順序か：
- **Geth（EL）が先に動いていないと**、LighthouseはJWT認証で接続先が見つからずエラーになります。
- **Lighthouse BN（CL）が先に同期を始めていないと**、VC（バリデータクライアント）はビーコンノードに接続できず、署名の判断材料を取得できません。

下位レイヤーから順に起動することで、各サービスが依存先に正しく接続できます。

```bash
# systemdの設定をリロードする
# 復元したサービスファイルをsystemdに認識させるために必要
sudo systemctl daemon-reload

# Geth（実行レイヤー）を起動する
sudo systemctl start geth

# 5秒待つ（GethのRPCエンドポイントが準備できるまでの時間）
sleep 5

# Lighthouse BN（合意レイヤー）を起動する
sudo systemctl start lighthouse

# 10秒待つ（Lighthouseがピアを見つけてビーコンチェーンへの接続を確立する時間）
sleep 10

# Lighthouse VC（バリデータクライアント）を起動する
sudo systemctl start lighthouse-vc
```

```bash
# 各サービスの起動確認
sudo systemctl status geth
sudo systemctl status lighthouse
sudo systemctl status lighthouse-vc
```

`Active: active (running)` が3つとも表示されていれば起動成功です。

## 5. 復元後のキャッチアップ過程（実際のログ）

サービスを起動しても、すぐに署名が再開されるわけではありません。
ノードが停止していた期間の分だけ、チェーンに追いつくまでの時間がかかります。

**今回の停止期間：約40時間46分（約1日16時間46分）**
**キャッチアップ所要時間：約3時間30分**

> 💡 **停止期間とキャッチアップ時間は別物です。**
> 今回は約41時間分の遅れを、約3時間30分で取り戻しました（約11.6倍の速度）。
> キャッチアップ時間は停止期間ではなく「同期距離（スロット数）」と
> 「ピア数・ネットワーク速度」によって決まります。

### Lighthouse VCのキャッチアップ

**起動直後のログ（`sudo journalctl -u lighthouse-vc -f` で確認）：**

```
Jun 10 23:16:54 INFO  All validators inactive  validators: 2, epoch: 101354, slot: 3243334
```

【読み方】
- `All validators inactive` : バリデータが「非アクティブ」状態です。**これはエラーではありません。**
- `validators: 2` : 管理しているバリデータが2個あることを認識しています。
- `epoch: 101354, slot: 3243334` : 現在のビーコンチェーンの位置。

**なぜ inactive になるのか：**

VCが起動してもビーコンノード（Lighthouse BN）がまだチェーンに追いついていない間は、VCは「どのスロットで何をすべきか」を判断できません。BNの同期完了を待っている正常な状態です。**inactiveが続いても故障ではありません。BNの同期完了を待つ正常な動作です。**

**約3時間30分後：**

```
Jun 11 02:45:06 INFO  Connected to beacon node(s)
Jun 11 02:45:06 INFO  All validators active
```

【読み方】
- `Connected to beacon node(s)` : VCがBNへの接続を確立しました。
- `All validators active` : 全バリデータがアクティブ状態になりました。この時点から署名が再開されます。

### Lighthouseの同期状況

**起動直後のログ（`sudo journalctl -u lighthouse -f` で確認）：**

```
Jun 10 23:19:18 INFO  Syncing  peers: "2", distance: "12247 slots (1 day 16 hrs)", est_time: "--"
```

【読み方】
- `Syncing` : 同期中です。
- `peers: "2"` : 現在2台のピアと接続しています。ピア数が増えるほど同期が安定します。
- `distance: "12247 slots (1 day 16 hrs)"` : 現在のチェーンの先端まで「12247スロット（約1日16時間分）」遅れています。
- `est_time: "--"` : まだ速度を計測中のため推定時間が出ていません。

```
Jun 10 23:20:30 INFO  Syncing  peers: "5", distance: "12153 slots", speed: "2.00 slots/sec", est_time: "1 hr 41 mins"
```

【読み方】
- `peers: "5"` : ピアが5台に増えました。
- `speed: "2.00 slots/sec"` : 1秒あたり2スロットの速度で追いついています。
- `est_time: "1 hr 41 mins"` : 約1時間41分で同期完了の見込みです。

> 💡 **同期中は何もしなくてよいです。** ログを眺めながら `distance` が減っていくのを確認するだけです。距離がゼロに近づくと `Synced` のログが出ます。

### Gethのキャッチアップ

**起動後のログ（`sudo journalctl -u geth -f` で確認）：**

```
Jun 10 23:19:28 INFO  Imported new chain segment  number=2,981,604  age=1d16h49m
```

【読み方】
- `Imported new chain segment` : 遅れていた分のブロックをまとめてインポートしています。
- `number=2,981,604` : 現在取り込んだブロック番号です。
- `age=1d16h49m` : このブロックが生成されたのは「1日16時間49分前」です。つまり現在のチェーン先端より約1日16時間遅れています。

この後、数分ごとに `number` が増加し、`age` が減少していきます。`age` が数分〜数十秒になれば同期完了が近いサインです。

### 署名再開の確認

```
Jun 11 02:47:13 INFO  Successfully published attestations  count: 1
```

【読み方】
- `Successfully published attestations` : アテステーション（署名）の送信に成功しました。
- `count: 1` : 1件のアテステーションを送信しました。

**このログが出て初めて「復元完了」と言えます。** サービスが起動しているだけでは不十分で、実際に署名が再開されるまでを確認してください。

## 6. 復元後の注意事項

### stale lockfile について

復元後のログに以下のような警告が出ることがあります：

```
WARN  Ignored stale lockfile
```

【読み方】
- `stale lockfile` : 「古くなったロックファイル」を検出しました。
- `Ignored` : Lighthouseが**自動的に無視しました**。

**なぜこれが出るのか：**

バックアップ時にサービスが稼働中だった場合、`/var/lib/lido-csm/` 内に `.lock` ファイルが含まれます。このファイルは「今このディレクトリを使っているプロセスがある」という目印として使われますが、復元先ではそのプロセスは存在しません。Lighthouseはこの矛盾を検出し、古いロックファイルとして自動的に無視します。

**手動での削除は不要です。** Lighthouseが正常に処理します。

### 本番環境（ベアメタル）での復元について

> 💡 **本章の手順はVM検証環境用です。本番環境（ベアメタル）での復元は一手間多くかかります。**

第2部のバックアップは、セキュリティリスクを考慮して**鍵本体を意図的に除外**しています。

| データ | 第4章バックアップ（VM用） | 第2部バックアップ（本番用） |
|---|---|---|
| slashing_protection.sqlite | ✅ 含む | ✅ 含む |
| validator_definitions.yml | ✅ 含む | ✅ 含む |
| validators/*/*.json（鍵本体） | ✅ 含む | ❌ 除外（セキュリティ上の理由） |
| keystore_password.txt | ✅ 含む | ❌ 除外（セキュリティ上の理由） |

鍵本体をバックアップに含めない理由は、バックアップファイルが漏洩した際のリスクを最小化するためです。バリデータ鍵は最初に生成した**ニーモニック（24単語）**から再生成できるため、鍵そのものをバックアップする必要はありません。

#### 本番環境での復元フロー

第2部のバックアップから復元する場合、以下の手順が追加で必要になります：

```
通常のバックアップ展開（serviceファイル・スラッシング保護DB等）
↓
Step A: ethstaker-deposit-cli でニーモニックから鍵を再生成
        （existing-mnemonic オプションを使用）
↓
Step B: lighthouse account validator import で鍵を再インポート
        （/tmp経由・第1部・第2部と同じ手順）
↓
Step C: lighthouse slashing-protection import でDBを復元
↓
Step D: サービス起動・署名再開確認
```

> ⚠️ **Step AとStep Bの詳細手順は第2部・Phase 4を参照してください。**
> ニーモニックは紙に手書きで保管しているオフラインのものを使用します。
> デジタルデータとして保存されたニーモニックは使用しないでください。

> 💡 **この一手間がセキュリティを守ります。**
> 鍵をバックアップに含めないことで、
> バックアップファイルが万が一漏洩しても
> バリデータ鍵は安全に保たれます。
> ニーモニックさえ手元にあれば、鍵はいつでも再生成できます。

### 旧環境の後始末

復元が確認できたら（`Successfully published attestations` を確認後）、念のため保存しておいた旧データを削除します。

```bash
# 復元確認後に旧バックアップデータを削除する
# rm -rf = ディレクトリを中身ごと強制削除する（元に戻せない）
# 必ず「Successfully published attestations」を確認してから実行すること
sudo rm -rf /var/lib/lido-csm.bak
```

> ⚠️ **`rm -rf` は取り消しできません。** 署名再開を確認した後にのみ実行してください。
