## 1. なぜこの章を書くのか

第6章でSSVノードの死活監視を拡張した際、「閾値署名の成功状況をどう確認するか」という課題に対して、最初は`docker logs`でログを読んで文字列を数える方法（ログ方式）を採用しました。しかしこれは実行に時間がかかり、ソフトウェアのバージョンが変わるとログの文言自体が変わってしまうリスクも抱えていました（第5章で実際に`Attestation published`→`Attestation sent`という文言変更が発生していた事例があります）。

その後、SSVノードが標準で提供しているPrometheus形式のMetrics API（`/metrics`エンドポイント）を調査したところ、同じ情報がはるかに軽量な数値として、最初から公式に用意されていることが分かりました。

この発見を機に、`node_check.sh`の[1]〜[11]セクション全体を振り返り、「どこがログ方式で、どこがすでにメトリクス（API）方式だったか」を整理しました。本章はその振り返りの記録です。

## 2. まずMetrics APIを覗いてみる

「メトリクス方式」と言われても、実際に何が公開されているのか分からなければ
活用できません。本章の議論を理解する前に、まず実際にSSVノードの
Metrics APIエンドポイントを覗いてみましょう。

```bash
# SSVノードが公開しているメトリクス全量を確認する
curl -s http://127.0.0.1:15000/metrics
```

> ⚠️ **出力は数百行に及びます。** Goランタイムの内部情報（`go_*`）、
> libp2p（P2P通信ライブラリ）の詳細統計、SSV独自の指標（`ssv_*`）まで、
> 非常に多くの項目が一度に表示されます。最初は圧倒されますが、
> 全部を読む必要はありません。

量が多いため、まずは「どんな指標が存在するか」の見出しだけを確認します。
Prometheus形式では、各指標の直前に`# HELP`という説明行が付いています。

```bash
# 指標名と説明（# HELP行）だけを一覧する
curl -s http://127.0.0.1:15000/metrics | grep "^# HELP"
```

これで、`ssv_`で始まる指標（SSVノード独自）や`ssv_p2p_`（P2P関連）、
`ssv_validator_`（バリデータ関連）といった分類が見えてきます。
気になる指標名が見つかったら、その名前で再度grepすれば、実際の値
（現在の値・付随するラベル）を確認できます。

```bash
# 特定の指標だけに絞って実際の値を確認する例
curl -s http://127.0.0.1:15000/metrics | grep '^ssv_validator_validators_per_status'
```

> 💡 **第6章・第7章で扱った`ssv_p2p_peers_connected`や、本章で扱う
> 各種指標は、こうして実際にエンドポイントを覗いて見つけたものです。**
> 「公式ドキュメントに書いてあったから」ではなく、「実際に動いている
> ノードが何を公開しているか、自分の目で確認した」という手順を、
> ぜひ読者自身の環境でも再現してみてください。

## 3. 「ログ方式」と「メトリクス方式」の違い

```
ログ方式：
  プロセスが出力する自由形式のテキスト（ログ）を読んで、
  特定の文字列パターンをgrepで探し、件数を数える

  例：docker logs ssv-node --since 1h | grep "100% success"
  例：journalctl -u lighthouse-vc -n 100 | grep "Successfully published"

メトリクス（API）方式：
  プロセスが標準で提供している、構造化された数値や状態を
  REST API・Prometheus形式のエンドポイントから直接取得する

  例：curl http://127.0.0.1:5052/eth/v1/node/syncing | jq '.data.is_syncing'
  例：curl http://127.0.0.1:15000/metrics | grep '^ssv_validator_validators_per_status'
```

### それぞれの特性比較

| 観点 | ログ方式 | メトリクス（API）方式 |
|---|---|---|
| 取得速度 | 累積ログ量に比例して遅くなる場合がある | 一定・高速（その時点の状態を即時返す） |
| 安定性 | ソフトウェアのバージョンアップで文言が変わると壊れる | エンドポイント名・指標名が変わらない限り安定 |
| 得られる情報の種類 | 「何が起きたか」の詳細（タイムスタンプ、個別の値など）に強い | 「今どういう状態か」の集計値・累積値に強い |
| 直感的な理解しやすさ | 高い（人間が読んでそのまま理解できる） | 慣れが必要（指標名やラベルの意味を把握する必要がある） |
| 向いている用途 | 初めて触るシステムの動作確認・トラブル時の詳細調査 | 継続的な死活監視・自動アラート・ダッシュボード化 |

### 補足：これはクラウド監視サービスと同じ設計思想

「メトリクス」という考え方自体は、Ethereumクライアント特有のものではありません。Azure MonitorやAWS CloudWatchのようなクラウド監視サービスでも、全く同じ構造が使われています。

| 概念 | Azure Monitor | AWS CloudWatch | Prometheus（今回のSSV） |
|---|---|---|---|
| 数値データの単位 | メトリクス | メトリクス | メトリクス |
| 「今の状態」を表す型 | ゲージ的なメトリクス | Gauge | gauge型（例：`ssv_cl_sync_status`） |
| 「積み上げていく回数」を表す型 | カウンター的なメトリクス | Count（Sum） | counter型（例：`ssv_runner_submissions_failed_total`） |
| 詳細イベントの記録 | Log Analytics（ログ） | CloudWatch Logs | アプリケーションのログ出力（`journalctl`等） |
| ダッシュボード | Azure Dashboard | CloudWatch Dashboard | Grafana |
| アラート | Azure Alerts | CloudWatch Alarms | Grafana Alerting / Alertmanager |

```
クラウド監視で言われる一般的な原則：
  「メトリクスは軽量で集計向き、ログは詳細だが重い」
  「Gaugeは現在値、Counterは増え続ける累積値」
  「アラートはメトリクスに対して設定するのが基本。
    詳細調査が必要な時だけログを掘る」

→ 第6章・本章セクション4で実際に体験した「ssv_runner_submissions
  （gauge型）とssv_runner_submissions_failed_total（counter型）を
  単純に並べて比較すると誤解を招く」という発見は、
  まさにクラウド監視の現場で必ず通る「メトリクスの型を
  意識する」という基本作法そのもの
```

> 💡 **Prometheus形式とクラウド監視サービスの違いは「提供形態」だけ。**
> Azure MonitorやCloudWatchは、クラウドプロバイダーが用意した
> 「サービス」として提供され、裏側の実装を意識する必要はありません。
> 一方Prometheus形式は、オープンソースの「規約」であり、
> アプリケーション自身（SSVノード等）が自発的にこの形式で
> メトリクスを公開しています。考え方・設計思想そのものは共通しているため、
> クラウド監視の経験がある方は、今回の話を「知っている構造の
> 再発見」として読めるはずです。

## 5. なぜ[5]だけがログ方式のままなのか

[12]の改修にあたり、同じ発想で[5]（Recent Attestations）も改修できないか検討しました。

```
[5]の現在の実装：
  journalctl -u lighthouse-vc --no-pager -n 100 \
    | grep "Successfully published attestations" | tail -n 3
```

これは直近のアテステーションの**詳細**（どのバリデータが・どのスロットで・どのブロックに対して署名したか）を表示する項目です。調査した結果、Lighthouse/Nimbusの標準APIには、この種の「直近の署名イベントの詳細」を返すエンドポイントが見当たりませんでした（提供されているのは`is_syncing`や`peer_count`のような状態確認系APIが中心）。

> 💡 **メトリクス方式が万能ではない理由がここにある。** メトリクス
> （Prometheus形式）は「カウンターの値」「現在の状態フラグ」のような
> **集計値**の表現に向いていますが、「誰が・いつ・何に対して署名したか」
> という**個別イベントの詳細**を表現するのには向いていません（それを
> やろうとすると、ラベルの組み合わせが爆発的に増えてしまう）。
> このような情報は、今のところログを読む以外に確実な取得方法がない、
> というのが正直な結論です。

### 判断基準（まとめ）

```
「今どういう状態か」を知りたい
  → まずメトリクス（API）方式を探す
    （is_syncing、sync_status、validators_per_status等）

「何が・いつ起きたか」の詳細を知りたい
  → ログ方式が現実的な選択肢になる
    （個別のアテステーション内容、エラーの詳細メッセージ等）
```

## 6. 死活監視の設計を振り返って分かったこと

### 5-1. 「最初からこうすればよかった」のか

第6章での発見を踏まえ、「最初からメトリクス方式で設計すればよかったのではないか」という疑問が生まれました。振り返ると、これは必ずしも正しい反省ではありません。

```
最初の段階（システムを初めて動かす時）：
  ログを読んで「動いているらしい」と確認するのが、
  一番直感的で、文脈（何が起きているか）も理解しやすい

運用が進んだ段階（継続的に監視したい時）：
  「同じ確認を繰り返す」「自動化したい」というニーズが生まれ、
  この時点でメトリクス方式の利点（速度・安定性）が活きてくる
```

つまり、**「ログ方式で動かし方を理解し、メトリクス方式で運用を磨く」という順序自体が、自然な成長過程**だったと考えられます。最初からメトリクスだけで設計しようとすると、「何が重要な指標か」を判断する材料（実際の動作の手触り）がない状態で設計することになり、逆に的を外したダッシュボードになるリスクがあります。

### 5-2. 公式ドキュメントが体系的に整備されていない領域

Ethereumホームステーキングの個人運用において、「死活監視のベストプラクティス」を体系的にまとめた公式ドキュメントは、まだ多くありません（第6章で確認したSSVのベストプラクティスも、公式ドキュメントの複数ページに分散していました）。

```
現実的な運用スタイル：
  - 各クライアント（Lighthouse・Geth・SSVノード等）が個別に
    提供するAPI・メトリクスを、運用者自身が発見・組み合わせる
  - Grafana + Prometheus の組み合わせは広く使われているが、
    「どの指標を見るべきか」は経験的に判断されることが多い
  - 既存のダッシュボードテンプレート（公式・コミュニティ作成）を
    流用しつつ、自分の構成（兼業構成、ハイブリッド構成等）に
    合わせて手を加えるのが実務的なアプローチ
```

本章・第6章・第7章で記録した「ログからメトリクスへの移行」「項目を増やしすぎない判断（第6章15-5）」「累積値とゲージ値の型の違いに気づいた経緯」は、こうした体系化されていない領域における、実際の試行錯誤の記録として価値があると考えています。

### 5-3. 今後の活用：Grafanaパネル化への土台

本章で整理した5つのメトリクス（`ssv_cl_sync_status`、`ssv_el_sync_status`、`ssv_validator_validators_per_status`、`ssv_runner_submissions_failed_total`、`ssv_validator_duty_rounds_changed_total`）は、`node_check.sh`での利用に加えて、そのままGrafanaのPromQLクエリとしても活用できます。これは死活監視強化の次のステップ（恒常的なダッシュボード化）への土台になります。

## 7. 【参考】`node_check.sh` [12]セクション最終形

本章の議論を踏まえて改修した、`node_check.sh`の[12]セクション全体を掲載します。第6章の構築時点（ログ方式）から、本章での発見を経てメトリクス方式に置き換えた最終形です。

```bash
# ==========================================
# [12] SSV Node (DVT) Status
# ==========================================
echo -e "\n${YELLOW}[12] SSV Node (DVT) Status${NC}"

# Dockerがインストールされており、ssv-nodeという名前のコンテナが存在するか確認
if command -v docker >/dev/null 2>&1 && docker ps -a --format '{{.Names}}' | grep -q "^ssv-node$"; then
    # 1. コンテナの稼働ステータス
    SSV_STATUS=$(docker inspect -f '{{.State.Status}}' ssv-node 2>/dev/null)
    if [ "$SSV_STATUS" == "running" ]; then
        echo -e " - Container : ${GREEN}active (running)${NC}"

        # 2. P2Pピア数の取得 (Metrics APIから抽出)
        SSV_PEERS=$(curl -m 3 -s http://127.0.0.1:15000/metrics | grep '^ssv_p2p_peers_connected' | awk '{print $2}')
        if [ -n "$SSV_PEERS" ]; then
            echo -e " - P2P Peers : ${CYAN}${SSV_PEERS}${NC}"
        else
            echo -e " - P2P Peers : ${RED}Error fetching metrics${NC}"
        fi

        # 3. 最新の同期ログ抽出 (DutyScheduler のスロット処理を監視)
        # 【Note】これはログ方式のまま残している唯一の項目。理由は
        # セクション5を参照（「直近のイベント詳細」に該当するため）。
        SSV_LATEST_SLOT=$(docker logs ssv-node --tail 100 2>&1 | grep "DutyScheduler" | grep "received head event" | tail -n 1 | awk -F'"slot": ' '{print $2}' | awk -F',' '{print $1}')
        if [ -n "$SSV_LATEST_SLOT" ]; then
            echo -e " - Sync Slot : ${GREEN}${SSV_LATEST_SLOT}${NC} (Following Chain Head)"
        else
            echo -e " - Sync Slot : ${YELLOW}Waiting for chain events...${NC}"
        fi

        # 4. CL接続先の表示（片肺運転の可視化）
        # 【Why】SSVノードのCL（Beacon）接続は外部RPC（Alchemy等）に依存する
        # ハイブリッド構成のため、無料枠の枯渇や接続障害に気づけるよう
        # 「今どこに繋がっているか」を毎回明示する。
        SSV_BEACON_ADDR=$(docker inspect ssv-node --format '{{range .Config.Env}}{{println .}}{{end}}' 2>/dev/null | grep '^BEACON_NODE_ADDR=' | cut -d'=' -f2-)
        if [ -n "$SSV_BEACON_ADDR" ]; then
            if [[ "$SSV_BEACON_ADDR" == *"127.0.0.1"* ]]; then
                echo -e " - CL Source : ${GREEN}Local${NC} (${SSV_BEACON_ADDR})"
            else
                # ホスト名だけを抜き出して表示（APIキー等の漏洩を防ぐ）
                SSV_BEACON_HOST=$(echo "$SSV_BEACON_ADDR" | awk -F/ '{print $3}')
                echo -e " - CL Source : ${YELLOW}External (${SSV_BEACON_HOST})${NC} (Single point of dependency)"
            fi
        else
            echo -e " - CL Source : ${RED}Error fetching config${NC}"
        fi

        # 5. Metrics APIから各種公式メトリクスを一括取得
        # 【Why】以前はログ全文をgrepして集計していたが、SSVノードは
        # Prometheus形式で必要な数値（CL/EL健全性・duty成功率・
        # バリデータ状態）を公式に提供している。docker logsの読み出し
        # コスト（コンテナの累積ログ量に依存し時間がかかる）を避けられる。
        SSV_METRICS=$(curl -m 3 -s http://127.0.0.1:15000/metrics)

        # 5-1. CL（Beacon）の健全性：接続先がsyncedか
        SSV_CL_SYNCED=$(echo "$SSV_METRICS" | grep '^ssv_cl_sync_status{.*ssv_cl_sync_status="synced"' | awk '{print $2}')
        if [ "$SSV_CL_SYNCED" == "1" ]; then
            echo -e " - CL Status : ${GREEN}synced${NC}"
        else
            echo -e " - CL Status : ${RED}not synced (check CL Source!)${NC}"
        fi

        # 5-2. EL（ローカルGeth）の健全性：SSVノードから見て ready か
        SSV_EL_READY=$(echo "$SSV_METRICS" | grep '^ssv_el_sync_status{.*ssv_el_status="ready"' | awk '{print $2}')
        if [ "$SSV_EL_READY" == "1" ]; then
            echo -e " - EL Status : ${GREEN}ready${NC}"
        else
            echo -e " - EL Status : ${RED}not ready (check Geth/--ws!)${NC}"
        fi

        # 5-3. バリデータの実働状態（active/attesting/participating）
        SSV_VAL_ATTESTING=$(echo "$SSV_METRICS" | grep '^ssv_validator_validators_per_status{.*status="attesting"' | awk '{print $2}')
        if [ "$SSV_VAL_ATTESTING" == "1" ]; then
            echo -e " - Validator : ${GREEN}attesting${NC}"
        else
            echo -e " - Validator : ${YELLOW}not attesting${NC}"
        fi

        # 5-4. 閾値署名（COMMITTEE duty）の失敗回数（累積）
        # 【Why】「動いている」ことと「署名タスクを完遂できているか」は別。
        # ssv_runner_submissions_failed_total（累積失敗数・counter型）で
        # 確認する。
        # 【Note】ssv_runner_submissions（gauge型）は「直近の試行状態」を
        # 示す値であり、counter型のfailed_totalと単純に並べて
        # 「attempts/failed」として比較すると型の違いにより誤解を招くため、
        # 累積失敗数のみを表示する（セクション3の補足参照）。
        SSV_SUBMIT_FAILED=$(echo "$SSV_METRICS" | grep '^ssv_runner_submissions_failed_total{.*beacon_role="ATTESTER"' | awk '{print $2}')
        if [ -n "$SSV_SUBMIT_FAILED" ]; then
            if [ "$SSV_SUBMIT_FAILED" == "0" ]; then
                echo -e " - Duty Failed: ${GREEN}${SSV_SUBMIT_FAILED} (cumulative)${NC}"
            else
                echo -e " - Duty Failed: ${RED}${SSV_SUBMIT_FAILED} (cumulative, check logs!)${NC}"
            fi
        else
            echo -e " - Duty Failed: ${YELLOW}No submission metrics yet${NC}"
        fi

        # 5-5. ラウンドチェンジ（コンセンサスのタイムアウト・不安定さの兆候）
        # 【Why】署名自体は成功していても、QBFTコンセンサスが一発で
        # 決まらずタイムアウトでラウンドが変わった回数。
        # 値が大きい・増え続ける場合はクラスター内の誰かの応答が
        # 遅い可能性がある。
        SSV_ROUND_CHANGES=$(echo "$SSV_METRICS" | grep '^ssv_validator_duty_rounds_changed_total{.*reason="timeout"' | awk '{sum+=$2} END{print sum+0}')
        echo -e " - Round Chg : ${CYAN}${SSV_ROUND_CHANGES:-0} timeout(s)${NC}"
    else
        echo -e " - Container : ${RED}${SSV_STATUS}${NC}"
    fi
else
    echo -e " - Container : ${YELLOW}Not installed or not running${NC}"
fi
```

### 各項目の補足解説

| 項目 | 取得元 | 何を確認しているか |
|---|---|---|
| Container | `docker inspect`（コマンド） | コンテナがRestartingループに陥っていないか |
| P2P Peers | `ssv_p2p_peers_connected`（メトリクス） | 外の世界と繋がっているか（孤立検知） |
| Sync Slot | `docker logs`（ログ・例外的に残存） | 最新のHead Event受信を抽出し、EL/CL連携を証明 |
| CL Source | `docker inspect`（コマンド） | CL接続先のホスト名（ローカル/外部）を表示。片肺運転の可視化 |
| CL Status | `ssv_cl_sync_status`（メトリクス） | CL接続先（Alchemy等）が本当にsyncedか |
| EL Status | `ssv_el_sync_status`（メトリクス） | SSVノードから見てローカルGethがreadyか |
| Validator | `ssv_validator_validators_per_status`（メトリクス） | バリデータが実際にattesting状態にあるか |
| Duty Failed | `ssv_runner_submissions_failed_total`（メトリクス・counter型） | 累積の署名提出失敗回数 |
| Round Chg | `ssv_validator_duty_rounds_changed_total`（メトリクス・counter型） | コンセンサスのタイムアウト回数（不安定さの兆候） |

> 💡 **Sync Slotだけがログ方式のまま残っている点に注目してください。**
> これは見落としではなく、セクション5で説明した判断基準（「個別イベントの
> 詳細」はメトリクスでは表現しにくい）に基づく、意図的な選択です。

## 8. 【実録インシデント】予期しない電源断からの復旧 — HTTP/RPCとWebSocketの非対称性

第6章セクション15-2・15-3では`node_safe_stop.sh`を使った計画的な停止・再起動を扱いました。
本節は、その対極にある**予期しない電源断**（作業中のブレーカー瞬断によるもの）
からの復旧記録です。意図しないインシデントでしたが、計画的な停止検証では
見えなかった重要な発見があったため、本章（死活監視の設計思想）の一部として
記録します。

**発生状況：**

```
電源工事に伴うブレーカー瞬断 → サーバー・ルーターともに予期せぬシャットダウン
（node_safe_stop.shを経由しない、完全に無計画な停止）
```

**復電直後（約数分後）の確認：**

```
[1] System Services Status
 - geth: active
 - lighthouse: active
 - mev-boost: active
 - lighthouse-vc: active

[3] Sync Status
 - is_syncing:    false
 - sync_distance: 0

[12] SSV Node (DVT) Status
 - Container : active (running)
 - P2P Peers : Error fetching metrics
 - CL Status : not synced (check CL Source!)
 - EL Status : not ready (check Geth/--ws!)
 - Validator : not attesting
```

**同じ瞬間に、Lido CSM側（geth/lighthouse）は既に完全復旧していたのに対し、
SSVノードのEL/CL接続はまだ確立できていませんでした。**

**約10〜13分後の再確認：**

```
[12] SSV Node (DVT) Status
 - Container : active (running)
 - P2P Peers : 5
 - CL Status : synced
 - EL Status : ready
 - Validator : attesting
```

> ✅ **発見：HTTP/RPC系の接続（Geth・Lighthouse間の通信）は瞬断からの
> 復旧が速いが、WebSocket系の接続（SSVノード→ローカルGethのEL接続、
> `ws://127.0.0.1:8546`）は復旧にラグが出る、という非対称性が実地で
> 確認できた。**

```
HTTP/RPC系（Geth・Lighthouseの相互連携）：
  リクエスト/レスポンス型のステートレスな通信
  → 接続が切れても、次のリクエストを送ればそれだけで再開できる
  → 「再接続」という特別な手続きがほとんど不要
  → 復電後、サービスがactiveになった瞬間からすぐ会話を再開できた

WebSocket系（SSVノード → ローカルGethへのEL接続）：
  持続的な接続（コネクション）を張り続ける通信方式
  → 一度切れると、単に「次のリクエスト」では済まず、
    再度ハンドシェイク（接続確立）からやり直す必要がある
  → 再接続のロジック（リトライ、バックオフ等）の実装次第で、
    復旧にラグが生じやすい
```

> 💡 **「同じ瞬間に電源が戻った」という1つの事象でも、プロトコルの性質
> （ステートレスなHTTP/RPC vs 持続接続のWebSocket）によって、復旧の
> 速度が構造的に異なることが、今回のインシデントから明らかになりました。**
> これは設計上の優劣の問題ではなく、それぞれの通信方式が持つ性質の違いです。
> 本章で扱った「ログ方式 vs メトリクス方式」と同じく、通信の仕組みの
> 違いを理解しておくことが、死活監視の設計・トラブル時の判断に役立ちます。

> ⚠️ **今回のラグはあくまで「停止期間が数分〜十数分と短かった」場合の
> 実測値です。** 第7章（未稼働検証）で扱っているような、数日間にわたる
> 長期停止の場合は、Geth/Lighthouse自体も大量のブロックデータを
> 再ダウンロード・再検証する必要が生じるため、復旧時間は停止期間に
> 比例して伸びると考えられます。今回の「速やかな復旧」は、短時間の
> 停止であったことと、ディスク上にデータが保持され続けていたことが
> 両方揃った結果であり、一般的に「ラグがあっても常に大丈夫」と
> 拡大解釈すべきではありません。

> 💡 **この検証は、`node_safe_stop.sh`を使わない、完全に無計画な停止
> だったからこそ見えた発見でもあります。** 計画的な停止検証（第6章15-2）
> ではSSVコンテナを最初に・geth/lighthouseを最後に止める順序を意図的に
> 設計していましたが、今回のような不意の電源断では、すべてのプロセスが
> 同時に・無秩序に停止します。意図しないインシデントを「学びの機会」
> として記録することの価値を、改めて実感する事例でした。

## 9. まとめ

| 知見 | 内容 |
|---|---|
| ログ方式とメトリクス方式の違い | ログは「何が起きたか」の詳細に強く、メトリクスは「今どういう状態か」の集計値に強い |
| `node_check.sh`の実態 | [1]〜[11]はすでにAPI/コマンドベースで設計されていた。ログ方式だったのは[12]（改修前）と[5]のみ |
| [5]がログ方式のまま残る理由 | 「個別の署名イベントの詳細」を返すメトリクスが標準では提供されていないため |
| 判断基準 | 「状態を知りたい」→メトリクスを探す。「詳細・経緯を知りたい」→ログが現実的 |
| 設計順序についての教訓 | 「最初はログで動かし方を理解し、運用が進んだらメトリクスで磨く」のが自然な成長過程であり、反省すべき遠回りではない |
| 公式ドキュメントの現状 | 死活監視のベストプラクティスは体系化されておらず、運用者が個別のAPI・メトリクスを発見して組み合わせるのが実情 |
